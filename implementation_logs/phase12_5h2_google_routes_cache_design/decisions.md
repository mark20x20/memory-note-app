# Phase 12.5H-2 Decisions

## Routes API を Cloud Functions 経由にする理由

Google Routes API の API キーをモバイルアプリのバンドルに含めるとキーが漏洩するリスクがある。
Cloud Functions 経由にすることで:
1. API キーを Secret Manager で管理できる（アプリには含まれない）
2. サブスク判定・権限チェック・生成回数制限をサーバー側で行える
3. モバイルアプリから直接 API を呼べないよう強制できる
4. 将来のレート制限・コスト管理を一元管理できる

既存の `enrichNotePlaces`（Places API）も同じ理由で Cloud Functions 経由にしており、
ルート API でも同じパターンを踏襲する。

## route_segments を Firestore にキャッシュする理由

1. **コスト削減**: 同じ区間・移動手段に対して毎回 Routes API を呼ぶと従量課金が発生する。
   一度生成して TTL 30日で保存することで、再訪時の API コストをゼロにできる。
2. **オフライン対応**: キャッシュがあればネットワーク不在でも過去のルートを表示できる（将来）。
3. **高速表示**: API レイテンシ（〜1秒）を省けるため、地図を開くたびに即時表示できる。
4. **失敗時の追跡**: 生成失敗したセグメントを `status: 'failed'` として記録し、
   再試行を明示的に管理できる。

## owner/editor のみ生成可能にする理由

ルート生成には Google Routes API のコストが発生する。
課金の責任を明確にするため、ノートに権限を持つメンバー（owner/editor）のみに限定する。
viewer はノートに編集権限がなく、コストを発生させる操作をすべきでない。
また、viewer が生成できると「誰かが生成したコストを誰が払うか」が不明確になる。

## viewer はキャッシュ閲覧のみとする理由

ルートのキャッシュ情報はノートのコンテンツの一部。
共有ノートを見ている viewer が、owner が作ったルートを見られるのは自然。
ただし生成（API コストの発生）は viewer には許可しない。
この境界が「閲覧はできるが変更はできない」という viewer の役割定義とも一致している。

## 無料ユーザーは直線ルートにする理由

- Google Routes API には料金が発生するため、全ユーザーに開放するとコストが青天井になる。
- 直線ルートは追加 API コストなしで提供できる最低限の機能として成立している。
- 無料ユーザーにも訪問順の流れは伝わる（Phase 12.5G-5 / 12.5H-1 で確認済み）。
- 有料プランとの明確な差別化によって、Premium の価値を示す。

## Premium でも forceRefresh を制限する理由

`forceRefresh: true` はキャッシュを無視してルートを再生成する。
TTL 30日以内でも強制的に API を呼び出すため、コストが高い。
Premium ユーザーでも 1日3回に制限することで:
1. バグや誤操作による無駄な API 呼び出しを防げる
2. コスト予測がしやすくなる
3. 悪意あるユーザーが大量生成するのを防げる

通常の生成（キャッシュなし区間のみ）は 1日10回まで許可し、
forceRefresh はその中でも3回以内に制限する。
