# Phase 12.5G-4 Decisions

## ノート詳細をプレビュー画面に寄せる理由

ノートを「作品としての思い出」として見返す画面に仕上げるため。
再推定・候補確認・フロー分割設定などの管理UIがあると、管理画面のように見えて、
見返す体験が損なわれる。
プレビュー画面は「読む」場所、編集画面は「整える」場所と役割を明確に分ける。

## 編集操作を edit 画面に寄せる理由

フロー作成・再作成・フロー分割設定などは、ユーザーが意図を持って実行する操作。
ノート詳細の閲覧中に誤って再推定してしまうリスクを減らす。
編集画面に寄せることで「この画面で編集する」という明確な文脈ができる。

## フロー閲覧画面を追加する理由

既存の `places/[placeGroupId]` は候補地図・候補一覧・手動入力フォームがあり、
実質的に「フロー編集画面」だった。
閲覧専用の画面を別ルートとして用意することで、ノート詳細からのタップが
「見返す行為」として自然な体験になる。
写真・時刻・場所・一言メモを見やすく表示し、前後フローへの移動もできる。

## places/[placeGroupId] をフロー編集画面として扱う理由

候補地図・候補選択・手動入力・一言メモ編集はすべて「場所を修正する」という
編集操作であるため、フロー編集画面として位置づけるのが自然。
フロー閲覧画面から「編集」ボタンで遷移させることで、
閲覧→必要なら編集というフローが明確になる。

## 未確認や再推定などの管理UIをプレビュー画面から減らす理由

「要確認」バッジの連発や再推定ボタンなどの管理UIは、
見返す体験よりも「まだ整理が必要」という印象を与える。
ノート詳細は整理が完了した後も頻繁に開かれる画面なので、
常時これらを表示するのは不適切。
確認・修正が必要な場合は、編集画面やフロー閲覧画面経由で行う。
