# Phase 12.5E-2 Places UI Polish — Decisions

## D1: 「候補を再取得」ボタンを本番UIから外す

**決定:** `__DEV__` フラグで開発時のみ表示する。本番ユーザーには非表示。

**Why:**
- 「再取得」は Places API コストを増やす（1PlaceGroup あたり最大2コール）
- 一般ユーザーにとって「再取得」が何をするか不明瞭
- 候補が間違っている場合は「手動入力」を使うのが正しい選択

**How to apply:**
- `__DEV__` チェックは Expo のビルドモードで切り替わる（development build = true, production = false）
- 開発テスト時はそのままボタンが表示される

---

## D2: カードタップではなく明示ボタンにする

**決定:** VisitedPlacesSection と places/index.tsx の場所カードを `View` に変更し、「候補を確認・変更」の明示ボタンを追加する。

**Why:**
- 実機テストでカードをタップすると候補確認画面に遷移するが、意図が伝わりにくい
- 「カードを押すと何かが変わる」という直感はモバイルUXで一般的ではない場合がある
- 確認・変更というアクションは意図が明確なラベルで示す方がユーザーが安心できる

**How to apply:**
- owner/editor のみ「候補を確認・変更」ボタンを表示
- viewer は確認済み場所のみ「詳細を見る」ボタンを表示

---

## D3: 候補番号を表示する

**決定:** candidates 配列（distanceMeters 昇順）でのインデックス+1を `#1`, `#2` のように左端の丸バッジで表示する。

**Why:**
- 複数候補があるとき「1番目の候補」「3番目の候補」のように参照できる
- 将来 Map SDK で番号付きピンを実装するとき、UIの番号と地図ピン番号を対応させる基盤になる
- DBには保存しない（表示用のみ）

---

## D4: カテゴリタグを各所に追加する

**決定:** VisitedPlacesSection・places/index・places/[placeGroupId] の各場所カードにカテゴリタグを表示する。

**Why:**
- 「レストラン・写真1枚」のように種別が視覚的に分かるとユーザーが候補を見分けやすくなる
- Google Places の `types` 配列は英語なので、日本語ラベルに変換して表示する
- 候補確認画面ではフィルタにもカテゴリを使うため、一貫性がある

---

## D5: カテゴリフィルタを追加する

**決定:** 候補確認画面（places/[placeGroupId].tsx）の上部に横スクロールのカテゴリチップを追加する。

**Why:**
- 候補が20件ある場合、全体を縦スクロールするのは時間がかかる
- レストランを探しているときに歯医者や学校を飛ばせるようにしたい
- 実装はシンプルな Array.filter のみ（フィルタロジックは `types` ベースのセット参照）

**How to apply:**
- 候補が0件のカテゴリチップは非表示（「すべて」は常に表示）
- フィルタ後0件の場合は「このカテゴリの候補はありません」メッセージを表示

---

## D6: 手動入力導線を上部に移動する

**決定:** 候補確認画面の手動入力ボタンを最下部から上部バナーへ移動する。

**Why:**
- 実機テストで、候補が多い（20件）場合に手動入力ボタンがスクロールの下の方に埋もれる
- 「候補が全部違う → 手動入力」という判断は候補を見る前に表示されている方が自然
- バナー形式で「候補にない場合」と説明文を添えることで意図が明確になる

---

## D7: priority/other 分類を廃止して distanceMeters 昇順統一表示にする

**決定:** 候補を「訪問候補」「その他の近隣」に分けるのをやめ、カテゴリフィルタを代替とする。

**Why:**
- priority/other の2分類は実装上の都合であり、ユーザーには「なぜ2グループ？」が分かりにくい
- カテゴリフィルタが代わりになる（レストランだけ見たい → レストランチップを押す）
- 距離順を崩すとユーザーが「なぜこれが先？」と思う可能性がある
- シンプルな方が保守しやすい

---

## D8: AIランキングを使わない

**決定:** OpenAI / その他 AI による候補リランキングは実装しない。

**Why:**
- このアプリの主目的はレストラン特定ではなく、写真を地図・時系列・旅順で整理すること
- AI の「それっぽい」回答は誤確定リスクがある
- 近い候補をユーザーが選ぶ UX の方が安全かつ正確
