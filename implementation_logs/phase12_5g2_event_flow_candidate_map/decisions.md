# Phase 12.5G-2 Decisions

## この日の流れを主役にする理由

ユーザーが求めているのは「何時にどこにいたか」を写真つきで振り返る体験である。
単なる場所名一覧（VisitedPlacesSection）より、
時刻・写真・場所名がセットで並ぶタイムラインの方がこの目的に合う。
Detail 画面では VisitTimelineSection を先に、VisitedPlacesSection（推定ボタン含む）を後にした。

## 候補確認地図と全体地図を分ける理由

地図は用途が2つある。
- **全体地図（map.tsx）**: 訪問イベントの場所を俯瞰する → PlaceGroup を表示
- **候補確認地図（[placeGroupId].tsx）**: 特定のイベントの場所候補を比較する → PlaceCandidate を表示

この2つを混在させると「どの地図に何が表示されているか」が分からなくなる。
全体地図は旅のルートを見るための地図、候補確認地図は正しい場所を選ぶための地図として分離した。

## candidate番号とevent番号を別物として扱う理由

- イベント番号 (#1, #2, #3): この日の流れ・地図ピン・Places一覧で共通
- 候補番号 (#1, #2, #3): 候補確認画面の地図ピンと候補リストで共通

それぞれの文脈で独立しており、DB に保存する必要はない。
配列 index + 1 で導出するため、順序変更にも自動対応する。

## photoPreviewURLsを追加する理由

この日の流れでは「この時間にこの写真を撮った」という関係を示したい。
coverPhotoURL は1枚だが、複数枚の写真があれば複数のサムネイルを並べることで
「このイベントには写真が複数ある」ことが一目で分かる。
最大3枚に絞ることで Firestore の書き込みサイズを抑える。

## 分割しきい値をユーザー選択にする理由

旅行・日常・食事・散歩では適切なしきい値が大きく異なる。
例:
- 同じ通り沿いの複数の店を別イベントにしたい → 30分/50m（細かく）
- 午後の観光全体を1イベントにしたい → 180分/120m（ゆったり）

固定値では多様なユースケースに対応できないため、
プリセット（細かく/標準/ゆったり）として選択肢を提供する。
数値を直接入力させると複雑すぎるためプリセット方式を採用。

## AIランキングを使わない理由

候補のランキング付けを AI で行うとコストとレイテンシが増大する。
場所特定の最終判断はユーザーが行う設計なので、
AI ランキングよりも「距離順で候補を出してユーザーが確認」の方がシンプルかつ低コスト。
