# Phase 12.5G-1 Decisions

## PlaceGroupを訪問イベントとして扱う理由

PlaceGroup は元々「GPSが近い写真のまとめ」だったが、
ユーザーが求めているのは「訪れた場所の旅順」である。
同じ場所に複数回行っても別イベントになるべき（時間差 90分超）し、
連続した移動は1イベントとしてまとめるべき。
PlaceGroup にイベント時刻・順序を持たせることで、
地図・タイムライン・Places一覧が一致した番号体系になる。

## 地図番号を候補番号ではなくイベント番号にする理由

候補20件は「1つのPlaceGroupに対する場所候補」であり、
地図上の別ピンとして表示するものではない。
地図ピンは「ユーザーが訪れた可能性のある訪問イベント」の位置を示すべき。
#1/#2/#3 がmap/places/timelineで一致することでUXが統一される。

## 時間 + 距離でイベント分割する理由

- 距離だけだと：同じ建物内で時間を空けて別日に行った写真が同一グループになる
- 時間だけだと：移動中の写真（距離が近い）が別グループになりすぎる
- 両方組み合わせることで「同じ訪問」を正確にグルーピングできる
- しきい値（80m / 90分）はヒューリスティックだが実用的な初期値

## startAt/endAt/sortOrderを追加する理由

- sortOrder: Firestore の複合インデックスを増やさずに時系列表示できる
- startAt: 時刻表示（"10:20"）と並び順の根拠になる
- endAt: 将来的な滞在時間計算・タイムライン表示に使える
- 既存 createdAt は「Firestore ドキュメント作成時刻」であり、写真撮影時刻とは別

## 候補20件を地図に直接出さない理由

候補20件は Places API の検索結果であり、
実際にそこへ行ったかどうかは確認されていない。
地図ピンとして出すと「行っていない場所がたくさん表示される」体験になる。
候補は候補確認画面（places/[groupId]）内でのみ表示する。

## AIランキングを使わない理由

候補のランキング付けを AI で行うとコストとレイテンシが増大する。
場所特定の最終判断はユーザーが行う設計なので、
AI ランキングよりも「近い順で候補を出してユーザーが確認」の方がシンプル。
Phase 12.5G-1 では場所候補の取得ロジックは変えず、イベント分割にのみ集中する。
