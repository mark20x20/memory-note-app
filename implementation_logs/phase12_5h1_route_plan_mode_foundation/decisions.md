# Phase 12.5H-1 Decisions

## 無料プランは直線破線ルートにする理由

直線ルートは追加APIコストなしで訪問順の流れを視覚的に伝えられる。
全ユーザーが訪問イベントの順番を地図上で把握できる最低限の体験として提供する。
課金なしで使える範囲を明確にし、有料機能との対比を作ることで価値を伝える。

## 有料プランは徒歩・車・公共交通ルートにする理由

実際の移動体験（歩いた道・乗った電車）を地図上で再現することで、
記録としての精度と体験の豊かさが増す。
Google Routes API は有料・認証が必要なため、プレミアム機能として分離することが自然。
移動時間・距離・乗換情報まで残せることで、有料の価値が明確になる。

## 今回Google Routes APIを実装しない理由

- 課金判定（RevenueCat / App Store Subscription）が未実装
- Google Routes API の API key 設定・課金設計が別途必要
- ルート結果の Firestore キャッシュ設計が未完了
- EAS Build + Android 確認が未済
- 今回は UI・型・将来拡張の土台作りに専念するため

## プレミアム誘導を控えめにする理由

記録アプリの主目的は「思い出を残す」こと。
課金プレッシャーが強すぎると、ユーザーが気持ちよくアプリを使えなくなる。
プレミアム機能は「こんなこともできるようになります」という提案にとどめ、
現在使える無料機能の価値を下げないようにする。

## 候補確認地図にはルート線を出さない理由

`places/[placeGroupId]` の候補確認地図は「この1つの場所の候補を比較する」画面。
複数候補ピンが並ぶ文脈でルート線を出すと、どのピンをつないでいるか混乱する。
また候補確認は「この場所が正しいか？」を判断する作業であり、
全体の訪問順とは別の文脈のため、ルートUIは不要。
