# Phase 12.5G-5 Decisions

## 写真フルスクリーンビューを追加する理由

「思い出を見返す」体験の中心は写真。
サムネイルのグリッドや小さなプレビューでは、写真が持つ記憶の質感・細部が伝わらない。
フルスクリーンで大きく表示することで、写真を見返す体験が豊かになる。

## Polyline を直線ルートにする理由

訪問イベント間の移動経路（道路・電車）を正確に再現するには Google Directions API などが必要。
しかし今回の目的は「どこからどこへ移動したか」の訪問順を地図上でひと目で分かるようにすること。
直線でつなぐだけでも、旅の流れ・方向感は十分に伝わる。
また Directions API は有料・レート制限・移動手段推定などの複雑さがあり、今回のスコープ外。

## Directions API を使わない理由

- 有料 API（Google Directions API）の課金リスク
- 移動手段（徒歩/車/電車）の推定が必要で実装コストが高い
- 正確なルートよりも「訪問順の可視化」が目的のため不要
- 今回は閲覧体験の向上が主目的であり、精度よりシンプルさを優先

## 候補確認地図には Polyline を出さない理由

`places/[placeGroupId]` の候補確認地図は、1つのフローの候補地点を比較するための地図。
複数の候補ピンを見比べる文脈に Polyline があると混乱を招く。
また候補地図は「全体の訪問順」ではなく「この場所の候補比較」に集中すべき。

## フロー閲覧画面から写真を開けるようにする理由

フロー閲覧画面（`flows/[placeGroupId]`）は「この場所の思い出を見返す」画面。
写真サムネイルをタップしても反応がないと、写真を大きく見たいユーザーが困惑する。
フロー単位の写真ビューアを提供することで、フローから写真への自然な遷移が生まれる。
PlaceGroupDoc の `photoIds` を使いそのフローの写真だけを絞り込むことで、
「このフローの写真」として文脈が明確になる。
