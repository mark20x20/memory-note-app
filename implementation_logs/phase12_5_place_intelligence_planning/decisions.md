# Phase 12.5 Place Intelligence Planning — Decisions

## D1: Phase 12.5 を Phase 13 の前に独立フェーズとして追加

**決定:** Phase 13（Search / Calendar）の前に Phase 12.5 を挿入し、場所推定・施設候補取得・ユーザー確認 UI を先に完成させる。

**理由:** 場所情報はAI日記・検索・カレンダー・共有カードすべての品質を支える基盤。Phase 13 が場所データを前提とするため、先に完成させる必要がある。

---

## D2: MVPプロバイダを Google Places API (New) に決定（実装前確認必要）

**決定:** 第一候補として Google Places API (New) を選定した。Nearby Search を使用し、Cloud Functions 経由で呼び出す。

**理由:** 日本国内での施設名・カテゴリの精度が最高水準。フィールドマスクによるコスト制御が可能。既存の Cloud Functions スタックとの相性が良い。

**注意:** 実装前に Foursquare との精度比較テスト・Google Maps ToS 最終確認が必要。

---

## D3: AI はゼロから場所を断定しない（候補補助のみ）

**決定:** OpenAI は外部 API の候補リストから最適候補を選ぶ役割に限定する。座標だけ渡して場所を推測させることは禁止。

**理由:** AI の hallucination リスク。候補外の場所が「確定」として保存されるのはデータ品質上致命的。

---

## D4: 場所推定は「ユーザーが明示的に実行した場合のみ」外部 API を呼ぶ

**決定:** ノート保存時に自動で Places API を呼ばない。Detail 画面の「場所を推定する」ボタンを押したときのみ実行する。

**理由:** プライバシー（位置情報の外部送信をユーザーが制御できるべき）・コスト（GPS なし写真のみのノートに無駄なAPI呼び出しを防ぐ）・UX（ノート保存速度を落とさない）。

---

## D5: Phase 12.5 を 6 つのサブフェーズに分割

**決定:** 12.5A（プロバイダ選定）→ 12.5B（データモデル）→ 12.5C（Cloud Functions）→ 12.5D（スコアリング）→ 12.5E（UI）→ 12.5F（Map SDK）の順に実装。

**理由:** 各サブフェーズが独立して完了・デプロイできる。データモデルとAPIが先に確定しないとUIが作れない依存関係を反映。

---

## D6: Phase 8 の MapPreview は「簡易版として完了」と再定義

**決定:** Phase 8 の React Native View ベースの MapPreview / 簡易グループ化は、「簡易版として完了」とみなす。Phase 12.5F で本格 Map SDK に置き換える。

**理由:** Phase 8 の成果物を無駄にしない。MapPreview は Phase 12.5F まで引き続き機能する。

---

## D7: 精密座標は共有カード・公開コンテンツに出力しない

**決定:** 共有カード・viewer 向け表示では、緯度経度の数値を出力しない。確定した場所名（例: 浅草寺）・エリアラベル（例: 東京 浅草）のみを表示する。

**理由:** 位置情報プライバシー。精密座標から自宅・職場・行動パターンが特定される可能性がある。

---

## D8: Places API キャッシュ期間を 7日間に設定

**決定:** Google Places API のレスポンスキャッシュは 7日間（ToS の上限 30日以内）。

**理由:** ToS 準拠の範囲で適度なキャッシュにより API コストを抑制する。7日は観光地・レストランの基本情報変更頻度と現実的にバランスが取れている。

---

## D9: viewer は Places 候補リストを閲覧不可

**決定:** viewer 権限のユーザーは確定場所名のみ閲覧可能。候補リスト・座標・rawデータは非表示。

**理由:** 外部 API の生データを最小限のユーザーにのみ見せる。プライバシー保護とコスト意識の観点。

---

## D10: PlaceGroup データモデルに userConfirmed フラグを必ず持つ

**決定:** PlaceGroupDoc に `userConfirmed: boolean` を必須フィールドとして持ち、ユーザーが確認するまでは `false` のまま。

**理由:** ユーザーが確認していない推定データを「確定情報」として扱わない。AI 日記・共有カードには `userConfirmed=true` の場所名のみを優先的に使用する。
