# Phase 9: OpenAI Prompt Design

## 1. 基本方針

- 写真画像は送らない。メタデータ（タイトル・メモ・撮影日時・写真枚数・場所グループ数）のみ使う
- 短文（100〜200 文字）の日本語の思い出日記を生成する
- 事実から推測できないことを創作しない（ハルシネーション防止）
- 丁寧すぎず、友人に話すような自然な文体

## 2. System Prompt 案

```
あなたは「思い出日記ライター」です。
ユーザーが入力したノートのメタデータ（タイトル・メモ・撮影日時・写真枚数・場所の数）をもとに、
短い思い出日記（100〜200文字）を日本語で書いてください。

ルール:
- メタデータに書かれていない具体的な場所名・人名・出来事を創作しないこと
- 天気・会話内容・料理の味など、入力に存在しない情報を断言しないこと
- 「〜だったかもしれない」「〜のような」などの推測表現を避け、確実なことだけを書く
- 写真の見た目・被写体の詳細を推測しないこと（画像は受け取っていない）
- ポジティブで温かいトーンを保つ
- 出力は日記本文のみ。前置き・説明文・タイトルは不要
- 100〜200文字に収める
```

## 3. User Prompt 案

```typescript
function buildUserPrompt(ctx: DiaryContext): string {
  const lines: string[] = [];

  lines.push(`【ノートタイトル】${ctx.title}`);

  if (ctx.memo) {
    // メモは長すぎる場合は切り詰める（個人情報の送りすぎを防ぐ）
    const truncatedMemo = ctx.memo.length > 200
      ? ctx.memo.slice(0, 200) + '…'
      : ctx.memo;
    lines.push(`【メモ】${truncatedMemo}`);
  }

  lines.push(`【ノートの種類】${ctx.noteType === 'shared' ? '共有ノート（複数人で作成）' : '個人ノート'}`);
  lines.push(`【写真の枚数】${ctx.photoCount}枚`);

  if (ctx.takenAtList.length > 0) {
    // 撮影日時の最初と最後だけ渡す（全件は不要）
    const sorted = [...ctx.takenAtList].sort();
    const first = sorted[0];
    const last = sorted[sorted.length - 1];
    if (first === last) {
      lines.push(`【撮影日】${first}`);
    } else {
      lines.push(`【撮影期間】${first} 〜 ${last}`);
    }
  }

  if (ctx.locationGroupCount > 0) {
    lines.push(`【訪れた場所の数（おおよそ）】${ctx.locationGroupCount}ヶ所`);
  }

  lines.push('');
  lines.push('上記の情報をもとに、短い思い出日記を書いてください。');

  return lines.join('\n');
}
```

## 4. 入力 Context 型

```typescript
type DiaryContext = {
  title: string;           // ノートタイトル（必須）
  memo: string;            // ユーザーメモ（空文字可）
  noteType: 'personal' | 'shared';
  photoCount: number;      // 写真枚数
  takenAtList: string[];   // 撮影日時リスト（ISO 8601 / "YYYY-MM-DD" 形式）
  locationGroupCount: number; // 場所グループ数（Phase 8 の groupNearbyLocations 結果）
};
```

## 5. 生成文体

| 項目 | 方針 |
|---|---|
| 文体 | 友人への手紙のような自然な日本語 |
| 人称 | 一人称（「私は」「〜した」）または三人称視点を避けた日記調 |
| トーン | ポジティブ・温かい |
| 禁止 | 「〜と思います」「〜かもしれません」などの推測表現 |
| 禁止 | 画像内容の推測（「青い空が広がっていた」など） |
| 禁止 | 入力にない場所名・人名・食べ物の具体的な記述 |

## 6. ハルシネーション防止方針

- **System Prompt でルールを明示する**: 創作・推測・断言禁止
- **メモの文字数制限**: 最大 200 文字に切り詰めてから送る
- **画像を送らない**: 画像がなければ画像内容の捏造は起きにくい
- **場所情報は「グループ数」のみ**: 具体的な緯度経度は送らない（座標から地名を捏造させない）
- **モデル選択**: gpt-4o-mini を使用。温度（temperature）は 0.7 に設定（過度な創作を抑制）

## 7. 生成例

### 入力例

```
タイトル: 箱根旅行
メモ: 温泉最高だった
写真: 18枚
撮影日: 2026-05-03 〜 2026-05-04
場所グループ: 3ヶ所
```

### 出力例（期待値）

```
2日間の箱根旅行。18枚の写真に残った思い出。3ヶ所を巡り、
温泉にも浸かった。ゆっくり流れた時間がいつまでも心に残りそうだ。
```

### 悪い出力例（避けるべき）

```
朝から晴れ渡る青空のもと、芦ノ湖を見下ろすホテルの露天風呂に入った。
（← 入力にない情報を創作している）
```

## 8. 文字数目安

- 最低: 80 文字
- 目安: 100〜200 文字
- 最大: 250 文字（`max_tokens: 300` でカバー）

短すぎても長すぎても Detail 画面の diaryCard に馴染まない。  
System Prompt で 100〜200 文字を明示する。

## 9. 写真内容を推測しない方針

Phase 9 では画像認識（Vision API）を使わない。理由：

- コスト（gpt-4o vision は gpt-4o-mini より約 5〜10 倍高い）
- プライバシー（ユーザーの写真内容を OpenAI に送ることへのユーザー説明コスト）
- Phase 9 の最小スコープ（文章生成の検証が先）

「写真の見た目」ではなく「写真が存在するという事実」と「メタデータ」から日記を生成する。  
例えば「18枚の写真」「3ヶ所を訪れた」「5月3〜4日」は事実として使える。

## 10. モデル選定

| モデル | 採用理由 |
|---|---|
| `gpt-4o-mini` | 日本語生成品質が十分・コスト低・レイテンシ小 |

- `gpt-4o` は品質は高いがコストが約 15 倍（Phase 15 でアップグレード検討）
- `gpt-3.5-turbo` は旧来モデル。gpt-4o-mini の方がコスト・品質ともに優れる
