# Phase 12.5: Place Intelligence / Location Enrichment — Overview

## このPhaseを追加する理由

Phase 8 では React Native View ベースの地図風プレビューと、約 220m 以内の写真を同一グループとみなす簡易ロジックを実装した。しかし現時点では、写真の GPS 座標から **「どこに行ったか」を名称・カテゴリとして推定する機能が存在しない**。

ノートには緯度・経度は記録されているが、「浅草寺」「代官山 蔦屋書店」「渋谷駅」といった **人間が意味をつかめる場所名** は一切保存されていない。

このギャップを埋めるのが Phase 12.5 の目的である。

---

## AI日記より優先される理由

AI 日記（Phase 9）は、写真の枚数・撮影日時・場所グループ数という **メタデータのみ** をもとに文章を生成している。このため、日記に出てくる「3か所を巡りました」のような記述は、実際にどこへ行ったかを読み手に伝えられない。

一方、場所名が「浅草寺・仲見世通り・東京スカイツリー」と確定していれば：

- AI 日記の内容が具体的かつ正確になる
- 共有カードに「東京 浅草」と表示できる
- 検索・カレンダー・タイムラインが「どこへ行ったか」軸で機能する
- 振り返りの価値が根本的に向上する

つまり、**場所推定は AI 日記の品質を下から支える基盤**であり、Phase 13 の検索・カレンダーが場所を使えるかどうかも、この Phase に依存する。

---

## アプリの目玉機能としての位置づけ

このアプリのコアバリューは、「**写真を選ぶだけで、どこへ行ったかが整理された思い出ノートができる**」体験にある。

- 写真 → 撮影日時・GPS は Phase 6 で取得済み
- 写真 → 地図ピン表示は Phase 8 で実装済み
- **写真 → 場所名・施設名の推定は未実装**（Phase 12.5 で実現）

この機能が揃ってはじめて、「アルバムアプリとの差別化」「地図付きノートとしての完成度」が成立する。

---

## 「住所変換」と「場所推定」の違い

### 住所変換（逆ジオコーディング）
- 緯度経度 → 住所文字列（例: 東京都台東区浅草 2-3-1）
- 精度は高いが、「浅草寺」という施設名は出てこない
- Google Geocoding API / Nominatim が担う

### 場所推定（Places候補取得）
- 緯度経度 → 周辺の施設候補リスト（例: 浅草寺、仲見世通り、SHIBUYA SKY）
- 施設名・カテゴリ（restaurant / tourist_attraction / station）・評価などが得られる
- Google Places API (Nearby Search) / Foursquare Places 等が担う

本 Phase では、**場所推定（Places 候補取得 + スコアリング）を中心に設計する**。住所変換は補助的に使用する。

## 写真GPSから何ができるか

| 入力 | 処理 | 出力 |
|---|---|---|
| 写真の latitude / longitude | 周辺施設候補を取得（Places API） | 候補リスト（名称・カテゴリ・距離） |
| 複数写真のグループ | 代表点を算出 → 候補取得 | グループ単位の場所名 |
| 候補リスト + ユーザーメモ | スコアリング + AI 補助 | 推定場所（confidence付き） |
| 推定場所 | ユーザー確認 / 手動修正 | 確定場所（userConfirmed=true） |
| 確定場所 | AI 日記・共有カード・検索に反映 | 体験品質向上 |

## 最初のMVPで実装する範囲（Phase 12.5）

| 機能 | 実装内容 |
|---|---|
| Places 候補取得 | Cloud Functions 経由で外部 Places API を呼び出し、周辺施設候補を取得 |
| 候補スコアリング | GPS距離・写真集中度・ユーザーメモ一致・カテゴリ優先度によるスコア算出 |
| AI 候補補助 | OpenAI に候補テキストを渡し、最適候補を推定・自然なラベルを生成 |
| ユーザー確認UI | 推定場所を Detail 画面に表示し、確認・修正・手動入力できる |
| PlaceGroup データ保存 | Firestore に PlaceGroup・PlaceCandidate を保存 |
| AI日記・共有カードとの連携 | 確定場所名を AI 日記プロンプトと共有カードに渡す |
| プライバシー制御 | 精密座標は共有しない、Places API には最小限の座標のみ送信 |

## 基本方針（必須）

```
写真に入っているのは多くの場合、住所ではなく緯度・経度である。
このPhaseでは緯度・経度から周辺の候補地を取得し、候補の中からレストラン・観光地・駅・地名などを推定する。
AIはゼロから場所を断定するのではなく、外部API候補とユーザーメモをもとに候補選定を補助する。
```

これは設計全体を通じて守るべき原則であり、AI が候補外の場所を勝手に作ること（hallucination）は許可しない。
