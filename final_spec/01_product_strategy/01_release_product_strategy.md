# Release Product Strategy v2

## 1. サービス概要

**Memory Note App / 思い出ノートアプリ** は、  
**写真を選ぶだけで、友人・家族・恋人との思い出を、地図付きノートとして共同で残し、SNSにも美しく共有できるクラウド型モバイルアプリ**です。

### ひとことで言うと
- 写真を整理するアプリではない
- 地図アプリでもない
- 日記アプリでもない
- 旅行予約やSNSフィードのアプリでもない

**写真を起点に、思い出の文脈を自動でノート化するアプリ**です。

### Release v1の位置づけ
Release v1 は、単なる検証版ではなく、**実際に公開し、日常的に使える本番アプリ**として設計します。  
そのため、以下を前提にします。

- クラウド保存あり
- アカウントあり
- 共有ノートあり
- 共同編集あり
- SNS共有カード必須
- 削除・権限・プライバシー説明あり

---

## 2. v2での方針変更

### 変更前
- ローカル保存中心
- アカウントなしまたは任意
- 共有は後回し
- 共同編集は後回し
- SNS共有は後回し

### 変更後
- **Release v1からクラウド保存前提**
- **Release v1からアカウント必須または強く推奨**
- **Release v1から写真アップロード**
- **Release v1から共有ノート対応**
- **Release v1から Owner / Editor / Viewer 権限対応**
- **Release v1から友人・家族・恋人との閲覧・編集対応**
- **Release v1からSNS共有カード生成を必須化**

### 方針変更の意味
このサービスは、そもそも**思い出が単独ではなく複数人で共有される**ことが多い領域です。  
旅行、カフェ巡り、記念日、家族のお出かけ、デート、友達との遊びは、1人の記録ではなく**共同体験の記録**として価値が出ます。

---

## 3. 解決する課題

### ユーザーが抱える課題
| 課題 | 具体例 |
|---|---|
| 写真は残るが、文脈が残らない | どこで撮ったか分からない |
| 思い出が人ごとに分散する | 自分、恋人、友人、家族で写真がバラバラ |
| 後から見返しにくい | 時系列や場所の流れが追えない |
| 日記を書くのが面倒 | 文章化のハードルが高い |
| 共有しても魅力が伝わりにくい | 写真だけでは一日の流れが伝わらない |
| 写真共有と記録が別々 | アルバム、地図、メモが分断される |

### サービスが解決すること
- 写真からその日の流れを自動整理する
- 地図と一緒に思い出を残せる
- 複数人で同じノートを育てられる
- SNSで見栄えよく共有できる
- 後から見返したときに「何があった日か」が分かる

---

## 4. 提供価値

### コア価値
**写真を、ただの画像から「意味のある思い出ノート」に変えること。**

### 主要な提供価値
| 価値 | 説明 |
|---|---|
| 自動整理 | 撮影日時・位置情報で写真を並べる |
| 文脈化 | どこで、いつ、誰と、何をしたかを残す |
| 共同編集 | 1人ではなく、関係性の中で思い出を作れる |
| 振り返りやすさ | 地図、日付、一覧、タグで見返しやすい |
| 共有しやすさ | SNSカードで美しく外部共有できる |
| 安心感 | 権限・削除・公開範囲が明確 |

### 競合と違う価値
- Google Photosのような写真保管中心ではない
- Apple PhotosのようなOS標準写真管理の代替ではない
- Google Mapsのような位置履歴プラットフォームではない
- Polarstepsのような旅行専用記録でもない

**写真から共同の思い出ノートを作る**ことに集中します。

---

## 5. 初期ターゲットユーザー

### 優先ターゲット
| ターゲット | 理由 |
|---|---|
| 旅行好きの若年〜ミドル層 | 写真枚数が多く、価値が伝わりやすい |
| カップル | 共同編集の需要が高い |
| 友人グループ | イベント単位で共同記録しやすい |
| 家族 | 子どもの記録、お出かけ記録と相性が良い |
| カフェ・グルメ巡りユーザー | 短時間の複数スポット記録と相性が良い |

### 日本市場で特に相性が良い層
- 旅行記録を丁寧に残したい人
- LINEで写真を送り合う習慣がある人
- Instagramに投稿する前に整理したい人
- 日記は続かないが、写真はたくさん撮る人

---

## 6. 主要ユースケース

### v1で強く想定するユースケース
1. **旅行の思い出を共同で残す**
   - 友人や恋人と同じノートを作る
   - 宿、食事、観光地、移動の流れをまとめる

2. **カフェ巡り・グルメ巡りを残す**
   - 店ごとに写真をまとめる
   - どこで何を食べたかを一目で見返せる

3. **デート記録を残す**
   - 2人の写真を地図付きで整理する
   - 記念日に共有カードとして出す

4. **家族のお出かけ記録**
   - 子どもの写真を家族で共有
   - 誰が何を撮ったかを1つのノートにまとめる

5. **友達とのイベント記録**
   - ライブ、花火大会、誕生日会などを共同ノート化
   - SNSカードで外部にも共有できる

---

## 7. 競合との差別化

### 差別化の軸
| 軸 | 本アプリ | 競合 |
|---|---|---|
| 写真起点 | 写真を選ぶだけでノート化 | 日記入力前提が多い |
| 地図付きノート | 一日の流れと場所が残る | 写真管理だけ、または地図だけ |
| 共同編集 | v1から対応 | 多くは後回し |
| SNS共有カード | v1必須 | 記録と共有が分断されがち |
| 日本向けUX | 日本語・日本の生活文脈に最適化 | 汎用設計が多い |
| 非公開中心 | 関係者内の思い出管理が中心 | 公開SNS化しやすい |

### 競合ごとの立ち位置
- **Google Photos**: 写真保存・検索の強者。代替しない
- **Apple Photos**: OS標準の写真体験。置き換えない
- **Google Maps**: 地図・ナビ・位置履歴の強者。置き換えない
- **Polarsteps**: 旅行専用記録の強者。競合しすぎない

### 本アプリが狙う空白
- 写真はある
- 場所もある
- でも、**関係者で一緒に編集できる思い出ノート**がない

ここを取ります。

---

## 8. クラウド保存・共同編集を入れる理由

### なぜクラウド保存が必要か
思い出は個人端末の中だけでは完結しないためです。

- 複数端末で見たい
- 機種変更しても残したい
- 友人や家族と共有したい
- 1人の端末障害で消えないようにしたい

### なぜ共同編集が必要か
思い出は、撮った人だけのものではないためです。

- 恋人が撮った写真もまとめたい
- 友人が撮った写真を同じノートに入れたい
- 家族で同じお出かけを記録したい
- 一人では抜ける文脈を補完できる

### v1での設計意図
共同編集を入れることで、最初からこのアプリを
**「自分の写真管理」ではなく「関係性の記録」**
として成立させます。

---

## 9. SNS共有カードを入れる理由

### 必須にする理由
SNS共有カードは、単なる飾りではなく、**アプリの成長導線**です。

- 思い出を美しく外へ出せる
- Instagram / LINE / X / Facebook で使える
- 見た人に「何のアプリ？」と伝わる
- 共有が新規流入につながる

### 共有カードで伝えるべき情報
- メイン写真
- タイトル
- 日付
- 場所名
- 短い日記
- ミニマップまたはルート感
- 写真枚数
- スポット数
- アプリ名 / ロゴ
- 「Made with Memory Note」ブランド

### 重要な意味
共有カードがあることで、アプリ内で作ったノートが
**閉じた記録**ではなく、**外へ広がる表現物**になります。

---

## 10. 初期リリースの価値提案

### Release v1 の一言価値提案
**写真を選ぶだけで、みんなとの思い出を地図付きノートにして、SNSでも美しく共有できる。**

### v1でユーザーが得るもの
- 撮った写真が整理される
- どこへ行ったか分かる
- 誰と何をしたか残せる
- 共同で編集できる
- 共有カードで見栄えよく出せる

### v1の完成条件
以下が成立していれば、Release v1 は本番価値があります。

- アカウント登録・ログインができる
- 写真をアップロードできる
- ノートをクラウド保存できる
- 共有メンバーを招待できる
- 権限を管理できる
- 共同編集できる
- SNS共有カードを生成できる
- 削除とプライバシー説明がある

---

## 11. サービスとしての方向性

### 方向性の定義
このサービスは、**写真管理アプリではなく、共同思い出ノートアプリ**です。

### 目指す体験
- 写真を入れる
- その日の流れが整う
- 地図で振り返れる
- 仲間と一緒に育てられる
- SNSでも綺麗に出せる

### 目指さない体験
- 公開SNSフィード
- いいね
- コメント
- フォロー
- 口コミ投稿
- ナビアプリ化
- 旅行予約アプリ化
- 写真バックアップアプリ化

---

## 12. リリース時に守るべき原則

### 必須原則
| 原則 | 内容 |
|---|---|
| 写真起点を崩さない | 文章入力起点にしない |
| 共同思い出を中心にする | 単独日記に寄せすぎない |
| 共有は美しく、でもSNS化しない | フィードやコメントは入れない |
| クラウドは必須だが、過剰保存しない | 必要なデータだけ扱う |
| 権限を明確にする | Owner / Editor / Viewer を必ず守る |
| 削除導線を明確にする | 何が消えるかを見せる |
| プライバシー説明を省略しない | 写真・位置情報・AI・共有を説明する |
| 競合の土俵に乗らない | Google Photos / Apple Photos / Maps / Polarsteps 代替を狙わない |

---

## 13. 判断が必要な論点

Release v1 を本番公開するにあたり、以下は人間承認が必要です。

### 要承認項目
| 論点 | なぜ重要か |
|---|---|
| 最終Release v1スコープ | 共同編集や共有範囲の線引きが必要 |
| クラウド基盤の選定 | **決定済み: Firebase**（`00_stack_migration_decision_react_firebase.md` 参照） |
| 画像保存ポリシー | 原本保存か圧縮保存かでコストとプライバシーが変わる |
| AIデータ送信ポリシー | どの情報を外部AIに送るかが重要 |
| 権限モデル詳細 | Owner / Editor / Viewer の操作範囲を確定する必要がある |
| 公開共有仕様 | 共有リンク、限定公開、端末内保存の扱い |
| SNS共有カード仕様 | 画像サイズ、ブランド表示、ルート表現の設計 |
| プライバシーポリシー | 本番公開の前提条件 |
| 利用規約 | 共同編集と削除ルールに直結 |
| ストア提出 | 審査対応が必要 |
| 本番デプロイ | 公開責任が発生する |

### 未確定のまま残すべき点
- 画像の原本をクラウドに保存するか
- ノートの公開範囲をどこまで許すか
- 共有カードにどこまで情報を載せるか
- AI生成文をどの程度自動化するか
- 共同編集の衝突解決をどの方式にするか

---

必要であれば次に、**この v2 方針を前提にした Release v1 の確定スコープ表（Must / Should / Won’t）** まで落として、実装可能なレベルに整理します。
