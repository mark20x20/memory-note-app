# Decision Table v2

## 1. 目的

Release v1 を**本番公開できる状態で実装開始する前に、必ず決めるべき事項**を整理する。  
特に以下を目的とする。

- 実装方針のブレを防ぐ
- プライバシー事故を防ぐ
- 共同編集・権限設計の破綻を防ぐ
- クラウド費用の暴騰を防ぐ
- ストア審査で詰まらないようにする

---

## 2. 前提となるRelease v1方針

| 項目 | 前提 |
|---|---|
| プロダクト | 写真を起点に、地図付き思い出ノートを自動生成するアプリ |
| 対象 | iOS / Android |
| 初期市場 | 日本市場優先 |
| 保存方式 | クラウド保存前提 |
| 認証 | アカウント前提 |
| 共有 | 共有ノートあり |
| 共同編集 | あり |
| 権限 | Owner / Editor / Viewer |

| 共有先 | Instagram / LINE / X / Facebook |

| AI | 日記ドラフト・タイトル生成のみを想定 |
| API方針 | 実装上の外部API利用は可。ただし低コスト優先 |
| AI APIキー | **OpenAI APIキーをモバイルアプリに直接入れない** |

## 4. クラウド基盤の意思決定

### 4.1 比較表

| 選択肢 | 向いているケース | 強み | 弱み | 推奨度 |
|---|---|---|---|---|
| Firebase | 個人開発、React Native + Expo、最短実装、認証・DB・ストレージを一体化したい | 実装が速い、React Native / Expo との相性が良い、Auth/DB/Storage/Functionsが揃う、リアルタイム実装が楽 | ベンダーロックイン、複雑クエリは設計注意、コストが読みづらい場合がある | **高（採用済み）** |
| Supabase | SQL中心、将来の移行性、関係データをしっかり扱いたい | Postgresで設計しやすい、SQLが強い、個人開発でも扱いやすい | リアルタイム・権限制御・Storageの設計を詰める必要 | 不採用（Firebase を維持） |
| Custom Backend | 独自要件が強い、長期的に自由度を最大化したい | 自由度が高い、将来拡張しやすい | 開発負荷が大きい、初期公開が遅れる、運用責任が重い | 低 |

### 4.2 推奨案

| 項目 | 推奨 |
|---|---|
| 初期推奨 | **Firebase** |
| 後で再検討 | Supabase への移行可能性を残す |
| 避ける案 | 初期から Custom Backend を自作する |

### 4.3 判断理由

| 理由 | 内容 |
|---|---|
| 個人開発との相性 | 少人数で短期間に本番公開するなら、認証・DB・ストレージ・関数をまとめて使える方が有利 |
| React Native + Expo との相性 | Firebase は React Native / Expo の情報量と実装例が多く、立ち上がりが速い |
| 写真保存 | 写真アップロード・サムネイル・派生画像の運用がしやすい |
| 共同編集 | 共有ノートのリアルタイム更新を作りやすい |
| 実装速度 | v1 の公開スピードを優先しやすい |

### 4.4 リスク

| リスク | 内容 | 影響 |
|---|---|---|
| ベンダーロックイン | 将来の移行コストが高くなる | 中 |
| コスト増 | 写真・転送・関数実行で費用が増える可能性 | 高 |
| 設計の複雑化 | 共同編集と権限管理を雑に作ると事故が起きる | 高 |
| 再設計リスク | Supabase に後から乗り換える場合、認証・DB・Storage を作り直す可能性 | 中 |

### 4.5 決定期限

| 項目 | 期限 |
|---|---|
| クラウド基盤選定 | **設計開始前** |
| バックエンド構成決定 | **DB設計着手前** |

## 5. 画像保存ポリシーの意思決定

### 5.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| 原本画像 | **原則保存しない、または条件付き保存** | 常時保存 | コスト・漏洩リスクを抑える | 高画質が必要な将来機能に制約 | DB/Storage設計前 |
| 圧縮画像 | **保存する** | 保存しない | 表示・共有・通信費の最適化 | 画質低下 | 同上 |
| サムネイル | **保存する** | 生成都度作成 | 一覧UXが良く、通信量が減る | 保存量増加 | 同上 |
| EXIF保持 | **最小限のみ保持** | EXIF完全保持 | 位置・端末情報漏えいを抑える | 解析精度低下 | 同上 |
| 削除範囲 | **DB・原本/圧縮・サムネイル・派生物を削除** | DBのみ削除 | プライバシー保護のため | 削除漏れリスク | 同上 |
| コスト影響 | **圧縮・サムネイル中心で抑制** | 原本中心 | 画像保存コストと転送量を抑える | 将来の再利用性低下 | 同上 |

### 5.2 推奨案

| 項目 | 推奨 |
|---|---|
| 初期推奨 | **圧縮画像＋サムネイル保存** |
| 原本画像 | **条件付き**（必要時のみ、または後回し） |
| EXIF | **位置情報など機微項目は最小化** |
| 削除 | **関連派生物まで完全削除** |

### 5.3 判断理由

- 画像が主データなので、保存コストが支配的になりやすい
- 原本を常時保存すると、漏洩時のダメージが大きい
- v1 は「見返せる体験」が重要で、原本品質は必須ではない
- EXIF をそのまま残すと、位置情報事故につながる

### 5.4 リスク

| リスク | 内容 |
|---|---|
| 原本保存によるコスト増 | ストレージ・転送・バックアップコスト増 |
| EXIF漏えい | 自宅や行動圏が推測される |
| 削除漏れ | サムネイルやキャッシュが残る |
| 品質不足 | 圧縮しすぎると満足度が落ちる |

### 5.5 決定期限

| 項目 | 期限 |
|---|---|
| 画像保存ポリシー | **実装前** |
| 削除範囲 | **実装前** |
| EXIF保持範囲 | **実装前** |

## 6. アカウント・認証の意思決定

### 6.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| メール認証 | **採用** | 不採用 | ベースの退避手段として必要 | 離脱率が上がる可能性 | 認証設計前 |
| Googleログイン | **採用** | 不採用 | Android・Web連携で便利 | Google依存 | 同上 |
| Appleログイン | **採用** | 不採用 | iOS審査・UX上重要 | 実装分岐増 | 同上 |
| 匿名ログイン | **非推奨** | 期間限定で許可 | 共有・保存・機種変更と相性が悪い | データ引継ぎ問題 | 同上 |
| アカウント削除 | **必須** | 後回し | ストア審査・法務・信頼のため | 実装漏れでリスク大 | 仕様確定前 |

### 6.2 推奨案

| 項目 | 推奨 |
|---|---|
| 初期推奨 | **メール + Google + Apple** |
| 匿名ログイン | **原則なし** |
| アカウント削除 | **必須** |

### 6.3 判断理由

- 共有ノート前提では、ユーザー識別が必須
- 機種変更や複数端末利用に耐える必要がある
- Apple/Google ログインは初期離脱を下げやすい
- 匿名ログインは後で共有機能と整合しづらい

### 6.4 リスク

| リスク | 内容 |
|---|---|
| ログイン摩擦 | 初回体験が重くなる |
| 実装分岐 | iOS/Android でログイン実装が増える |
| 削除対応負荷 | アカウント削除時に関連データも消す必要がある |

## 7. 共有ノート・権限の意思決定

### 7.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| Owner | **採用** | なし | 最終責任者が必要 | 所有者不在問題 | 権限設計前 |
| Editor | **採用** | Viewerのみ | 共同編集価値の中心 | 誤編集・削除事故 | 同上 |
| Viewer | **採用** | なし | 閲覧専用の安全な共有が必要 | できることが少なく感じる可能性 | 同上 |
| 招待方法 | **招待リンク + メールを優先** | QRのみ / 手動追加 | 実装とUXのバランスが良い | リンク漏えい | 同上 |
| メンバー削除 | **Ownerのみ可** | Editor可 | 事故防止 | 管理負担増 | 同上 |
| Owner退会 | **Owner移譲を必須** | 自動削除 | ノート消失を防ぐ | 実装が複雑 | 同上 |
| ノート削除権限 | **Ownerのみ** | Editorにも許可 | 誤削除防止 | Editorの自由度低下 | 同上 |
| 編集衝突 | **後回しにしつつ簡易上書き防止** | 完全CRDT/差分管理 | v1では複雑化を避ける | 同時編集の競合 | v1.1以降再検討 |

### 7.2 推奨案

| 項目 | 推奨 |
|---|---|
| ロール | **Owner / Editor / Viewer** 固定 |
| 招待 | **招待リンク + メール** |
| 削除 | **Ownerのみ** |
| Owner退会 | **移譲必須** |
| 編集衝突 | **初期は簡易対策、詳細管理は後回し** |

### 7.3 判断理由

- 共有アプリで一番危ないのは「誰が何をできるか不明」なこと
- 削除を Editor に与えると事故率が上がる
- Owner 不在は運用不能になる
- 複雑な編集衝突解決は v1 の開発負荷を押し上げる

### 7.4 リスク

| リスク | 内容 |
|---|---|
| 権限漏れ | Viewer が編集できる等の事故 |
| 削除事故 | ノートが消える |
| 所有者不在 | ノート管理不能 |
| 同時編集衝突 | 編集内容の上書き |

### 7.5 決定期限

| 項目 | 期限 |
|---|---|
| 権限モデル | **設計開始前** |
| 招待方法 | **画面設計前** |
| Owner退会ルール | **実装前** |
| 削除権限 | **実装前** |

## 8. AI送信データの意思決定

### 8.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| 写真本体を送るか | **原則送らない** | 送る | プライバシーとコストを抑える | AI精度が下がる可能性 | AI実装前 |
| サムネイルを送るか | **原則送らない、必要時のみ** | 送る | 最低限の見た目情報で十分な場合が多い | 情報不足 | 同上 |
| メタデータのみ送るか | **基本は送る** | 送らない | タイトル・短文生成の基礎になる | 推定精度限界 | 同上 |
| 場所名・日時・写真枚数 | **送る** | 一部のみ送る | 日記生成の中心情報 | 位置情報の扱いに注意 | 同上 |
| 共同編集メモ | **送らないか、明示同意時のみ** | 送る | 他人の記述を外部送信するリスク | 共同利用の信頼低下 | 同上 |
| AI生成回数制限 | **あり** | 無制限 | コスト制御のため | UX低下 | 同上 |
| バックエンド経由 | **必須** | アプリ直送 | APIキー保護のため | 実装増 | 同上 |

### 8.2 推奨案

| 項目 | 推奨 |
|---|---|
| AI送信データ | **メタデータ中心** |
| 写真本体 | **原則送らない** |
| サムネイル | **必要時のみ** |
| 共同編集メモ | **原則送らない** |
| 回数制限 | **必須** |
| 実行経路 | **Cloud Functions / Edge Functions / 自前バックエンド経由** |

### 8.3 必須明記事項

- **OpenAI APIキーをモバイルアプリに直接入れない**
- AI送信対象をユーザーに説明する
- 生成文は必ず編集可能にする

### 8.4 リスク

| リスク | 内容 |
|---|---|
| 機微情報流出 | 写真・位置・文脈の外部送信 |
| コスト増 | 生成回数に応じて費用増 |
| 誤生成 | 事実誤認の文章生成 |
| キー漏洩 | アプリ直埋め込みは危険 |

### 8.5 決定期限

| 項目 | 期限 |
|---|---|
| AI送信範囲 | **AI機能設計前** |

## 9. 地図・位置情報の意思決定

### 9.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| Google Maps SDK | **候補** | MapKitのみ | 表示品質・情報量が強い | コスト・利用規約 | 地図設計前 |
| MapKit | **候補** | Google Mapsのみ | iOS相性が良い | Androidは別実装 | 同上 |
| OpenStreetMap系 | **候補** | 商用SDK | 低コストで始めやすい | 表示品質や運用差 | 同上 |
| Geocoding API | **限定利用** | 使わない | 場所名推定に有効 | 呼び出しコスト増 | 同上 |
| 位置情報保存粒度 | **エリア単位優先** | 緯度経度完全保存 | 公開事故を減らす | 精度低下 | 同上 |
| SNSカード上の位置表示 | **位置ぼかし or 地名のみ** | ピン精密表示 | 外部公開事故を防ぐ | 表現力低下 | 同上 |
| 常時GPSなし | **必須** | 常時取得 | プライバシー・バッテリー保護 | 履歴精度低下 | 仕様確定前 |

### 9.2 推奨案

| 項目 | 推奨 |
|---|---|
| 地図SDK | **Firebaseではなく地図は別途、Google Maps / MapKit / OSMを比較して決定** |
| 初期推奨 | **OS標準・低コスト優先** |
| Geocoding | **必要最小限** |
| 位置保存 | **精密座標は内部限定、外部表示は弱める** |
| 常時GPS | **使わない** |

### 9.3 判断理由

- v1 の本質は「地図表示」ではなく「思い出ノート化」
- 位置情報はセンシティブなので、外部公開に慎重であるべき
- 常時GPSはバッテリー・権限・審査の負担が大きい

### 9.4 リスク

| リスク | 内容 |
|---|---|
| 位置特定 | 自宅・勤務先が推測される |
| APIコスト | Geocodingが写真数に比例して増える |
| 審査リスク | 位置情報利用説明が不十分だと問題化 |
| UX劣化 | 地図が弱すぎると価値が伝わらない |

### 9.5 決定期限

| 項目 | 期限 |
|---|---|
| 地図SDK | **画面設計前** |
| Geocoding方針 | **実装前** |
| 位置情報保存粒度 | **プライバシー設計前** |

## 10. SNS共有カードの意思決定

### 10.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|

| 表示項目 | **タイトル・日付・場所名・短文・写真枚数・スポット数・ロゴ** | 地図やルートを増やす | 情報量と安全性のバランス | ごちゃつき | 同上 |

| 位置ぼかし | **必須** | なし | 外部公開事故防止 | 表現力低下 | 同上 |
| 共有前プレビュー | **必須** | 任意 | 誤公開防止 | 工数増 | 同上 |
| SNS投稿導線 | **OS共有シート中心** | 個別連携実装 | 低コストで実装しやすい | 導線の弱さ | 同上 |

### 10.2 推奨案

| 項目 | 推奨 |
|---|---|
| 比率 | **1:1 / 4:5 / 9:16** |
| 表示情報 | **必要最小限 + ブランド** |
| 地図 | **ぼかし・簡略表示** |
| 位置情報 | **外部公開では抑制** |
| プレビュー | **必須** |
| 投稿導線 | **OS共有シート中心** |

### 10.3 判断理由

- SNS共有カードは成長導線だが、位置情報事故の温床にもなる
- 最初から細かいSNS別最適化をしすぎると開発が重い
- まずは「見栄えの良い共有画像」を安定して出すことが重要

### 10.4 リスク

| リスク | 内容 |
|---|---|
| 位置漏えい | 地図・住所・ルートから特定される |
| ブランド崩れ | 文字詰め・余白不足で見栄えが悪い |
| 実装負荷 | 比率ごとのレイアウト差分が増える |

### 10.5 決定期限

| 項目 | 期限 |
|---|---|
| カード比率 | **デザイン着手前** |
| 表示項目 | **デザイン着手前** |
| 位置ぼかし | **デザイン着手前** |

| AI | **回数制限あり** |
| ストレージ | **上限あり** |
| 有料化 | **後回し** |
| 監視 | **必須** |

### 11.3 判断理由

- v1 は価値検証が中心であり、いきなり課金設計を複雑化しない方がよい
- ただし画像・AI・地図はコストが積み上がるため、制御は必要
- 無料でも、将来の有料化を見据えた構造は残すべき

### 11.4 リスク

| リスク | 内容 |
|---|---|
| 赤字化 | 利用増に対して費用が増える |
| UX低下 | 制限が厳しすぎると使いづらい |
| 未収益化 | 価値が出ても収益回収が遅れる |

### 11.5 決定期限

| 項目 | 期限 |
|---|---|
| 料金方針 | **開発着手前** |
| AI制限 | **実装前** |
| ストレージ上限 | **実装前** |

## 12. プライバシー・法務の意思決定

### 12.1 比較表

| 項目 | 推奨案 | 代替案 | 判断理由 | リスク | 決定期限 |
|---|---|---|---|---|---|
| プライバシーポリシー | **必須** | 後回し | ストア審査と法務対応に必要 | 公開停止リスク | 公開前 |
| 利用規約 | **必須** | 後回し | 削除・権限・責任範囲の明示が必要 | トラブル拡大 | 公開前 |
| 共有ノートの責任範囲 | **明文化** | 曖昧なまま | 共同編集トラブルを防ぐ | 争いの原因 | 公開前 |
| 写真削除 | **完全削除方針** | 一部残す | プライバシー保護 | 復旧困難 | 実装前 |
| 位置情報削除 | **必須** | 任意 | 位置漏えい対策 | 住所特定 | 実装前 |
| アカウント削除 | **必須** | 後回し | 審査・信頼・法務上必要 | データ残留 | 公開前 |
| ストア審査文言 | **事前に固定** | 後で修正 | 申請不備を防ぐ | 審査落ち | 提出前 |

### 12.2 推奨案

| 項目 | 推奨 |
|---|---|
| 規約 | **公開前に確定** |
| ポリシー | **公開前に確定** |
| 写真削除 | **関連派生物含めて削除** |
| 位置情報削除 | **必須** |
| アカウント削除 | **必須** |

### 12.3 判断理由

- 写真・位置情報・共同編集はトラブル時の影響が大きい
- ストア審査でも、説明不足はリスク
- 削除導線は信頼性に直結する

### 12.4 リスク

| リスク | 内容 |
|---|---|
| 審査落ち | 権限・写真・位置情報の説明不足 |
| 法務トラブル | 削除・責任範囲の不明確さ |
| 信頼低下 | 消せない、公開範囲が分からない |

### 12.5 決定期限

| 項目 | 期限 |
|---|---|
| 規約・ポリシー | **実装前〜公開前** |
| 共有責任範囲 | **実装前** |
| 削除仕様 | **実装前** |
| ストア文言 | **提出前** |

## 13. Not Decide Now / 後回しにする項目

| 項目 | 後回し理由 |
|---|---|
| 詳細な編集衝突解決 | v1では複雑で、まずは簡易対策で足りる可能性が高い |
| 高度な検索条件 | ノート数が増えてから価値が出やすい |
| フォトブック | 物流・決済・印刷対応が重い |
| 動画自動生成 | コスト・品質リスクが高い |
| 限定共有リンクの細かな期限設計 | v1では招待リンクで代替可能 |
| 旅行計画機能 | 現在の中心軸から外れる |
| 常時GPS履歴 | バッテリー・プライバシー負荷が大きい |
| 人生マップ | 蓄積前提で、初期価値が薄い |
| 都道府県マップ | 面白いがコアではない |
| カフェ巡り特化テンプレ大量追加 | v1で必須ではない |
| SNSフィード・いいね・コメント・フォロー | 明確に非対象 |
| 公開プロフィール | プライバシーリスクが高い |
