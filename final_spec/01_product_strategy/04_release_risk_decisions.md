# Release Risk Decisions v2

Release v1 を「クラウド前提・共有前提・SNS共有前提の本番アプリ」として出すなら、**機能を作る前に意思決定すべき論点**が多いです。  
以下、**決めるべき事項 / リスク / 未決定事項 / 次アクション**を整理します。

---

## 1. リリース前に決めるべき重要事項

### 1-1. まず人間承認が必要な決定
Release v1 で以下は、実装前に確定が必要です。

| 項目 | 決めること | 理由 |
|---|---|---|
| クラウド方針 | **決定済み: Firebase を採用**（`00_stack_migration_decision_react_firebase.md` 参照） | 認証・DB・ストレージ・権限設計が変わる |
| 画像保存ポリシー | 原本保存するか、圧縮画像のみ保存するか | コスト・プライバシー・削除仕様に直結 |
| 共有ノート権限 | Owner / Editor / Viewer の権限範囲を最終確定 | 共同編集の事故防止 |
| 削除権限 | 誰が何を削除できるか | データ消失・トラブル防止 |
| 招待方法 | 招待リンク / メール招待 / QR の採用範囲 | 漏洩リスクとUXに影響 |
| 位置情報公開仕様 | 地図ピンをどこまで公開するか | センシティブ情報のため |
| AI送信範囲 | OpenAIへ何を送るか | プライバシー・説明義務に直結 |
| SNS共有カード仕様 | 地図、位置、短文、写真数などを何まで入れるか | 公開時の事故防止 |
| 課金方針 | v1で課金するか、無料で出すか | 依存サービスの選定と規模感に影響 |
| 公開範囲 | 共有ノートを初期から限定共有にするか | ストア審査・運用負荷に影響 |

### 1-2. Release v1 の採否判断
以下は「入れる/入れない」を決める必要があります。

| 論点 | 選択肢 | 推奨方向 |
|---|---|---|
| クラウド保存 | 必須 / 任意 / 後回し | **必須** |
| アカウント | 必須 / 任意 / なし | **必須または強く推奨** |
| 共同編集 | v1 / v1.1 / Phase 2 | **v1** |
| SNS共有カード | v1 / v1.1 / Phase 2 | **v1** |
| 公開フィード | v1 / なし | **なし** |
| 常時GPS | v1 / なし | **なし** |
| 原本画像保存 | v1 / なし / 条件付き | **条件付き**が現実的 |
| 位置情報の外部送信 | あり / なし / 条件付き | **条件付き** |
| AI利用 | 必須 / 代替可 / なし | **必須だがバックエンド経由** |

## 2. クラウド保存リスク

### 2-1. 主要リスク
Release v1 からクラウド保存前提にすると、以下がリスクになります。

| リスク | 内容 | 影響 |
|---|---|---|
| プライバシーリスク | 写真・位置情報・メモがクラウドに保存される | 高 |
| 削除漏れ | ノート削除時に画像・メタデータが残る | 高 |
| 権限漏れ | 本来見えないメンバーにデータが見える | 高 |
| 事故復旧難 | 誤削除・上書き時の回復が難しい | 中〜高 |
| コスト増 | ストレージ、通信、DB、転送量が増える | 高 |
| 同期不整合 | 複数端末・複数メンバーで状態がズレる | 中〜高 |

### 2-2. 特に重要な意思決定
| 決めること | 選択肢 | リスク |
|---|---|---|
| 原本保存 | 保存する / しない / 条件付き | 保存するとコストと責任が増える |
| 圧縮方針 | 常に圧縮 / 原本保持 / 使い分け | 画質と保存費用のトレードオフ |
| サムネイル | 別保存するか | UXは良くなるが保存量が増える |
| 削除範囲 | DBのみ / 画像も / バックアップも | 利用規約・説明に直結 |
| 保持期間 | 無期限 / 期間限定 / ユーザー選択 | 運用コストに影響 |

### 2-3. 推奨判断
- **v1 は圧縮画像＋サムネイル中心**が現実的
- 原本は「必要な場合のみ」または「将来対応」にする
- 削除は **DB、画像、派生画像の全削除** を原則にする
- ただしバックアップ領域の扱いは、実装前に必ず定義する

---

## 3. 共同編集・権限リスク

### 3-1. 主要リスク
共同編集は、v1 の価値である一方で最も事故が起きやすいです。

| リスク | 内容 | 影響 |
|---|---|---|
| 権限誤設定 | Viewer が編集できる、Editor が削除できる等 | 高 |
| 共有範囲の誤解 | 誰が何を見られるか分かりづらい | 高 |
| 削除事故 | 1人の操作でノートが消える | 高 |
| 退会後の扱い不明 | メンバー離脱後の写真・メモの扱いが曖昧 | 高 |
| 所有者不在 | Owner が退会・削除した場合の管理不能 | 中〜高 |

### 3-2. Owner / Editor / Viewer の責任範囲
必ず明文化すべきです。

| ロール | できること | できないこと |
|---|---|---|
| Owner | ノート削除、メンバー管理、権限変更、全編集 | なし、ただし法的責任は最大 |
| Editor | 写真追加、本文編集、場所名修正、メモ追加 | メンバー管理、ノート削除は不可推奨 |
| Viewer | 閲覧のみ | 編集、削除、招待変更 |

### 3-3. 削除権限の意思決定
| 論点 | 候補 | 推奨 |
|---|---|---|
| Editor に削除権限を与えるか | 与える / 与えない | **与えない** |
| Viewer のスクショ/転載制限 | できない / 注意表示のみ | **注意表示のみ** |
| Owner 退会時の扱い | 最後のOwner移譲 / 自動削除 | **移譲必須** |
| メンバー退会時の履歴 | 残す / 消す | **残すが可視性制御**が望ましい |

### 3-4. 推奨判断
- **ノート削除は Owner のみ**
- **Editor は編集のみ**
- **Viewer は閲覧のみ**
- Owner 不在は運用不能になるので、**Owner 移譲フローを必須**にする

## 4. 写真データリスク

### 4-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 個人情報化 | 写真そのものが個人情報・機微情報になり得る | 高 |
| 原本漏洩 | クラウドに高画質原本が残ると漏洩影響が大きい | 高 |
| 共有範囲漏れ | 共有ノートから本来見せたくない写真が見える | 高 |
| メタデータ漏れ | EXIF に位置情報や撮影端末情報が残る | 高 |
| 削除不完全 | サムネイルやキャッシュが残る | 高 |

### 4-2. 必ず決めること
| 決めること | 重要点 |
|---|---|
| 保存する画像の種類 | 原本 / 圧縮 / サムネイル |
| EXIF の扱い | 保存するか、削除するか、用途限定か |
| 共有時の画像品質 | 元画像そのままか、再圧縮するか |
| 端末キャッシュ | いつ消すか |
| 削除時の範囲 | 画像、派生画像、AI生成文、メタデータ |

### 4-3. 推奨判断
- **共有用と保存用で画像を分ける**
- **EXIF は保存最小化**
- 共有カードや外部共有に **元画像をそのまま出さない選択肢** を検討
- 削除時は **派生物も含めて消す** 方針が安全

---

## 5. 位置情報リスク

### 5-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 住所特定 | 自宅・勤務先・よく行く場所が推測される | 高 |
| 生活圏露出 | 位置履歴から行動パターンが見える | 高 |
| 共有カード漏洩 | SNSに出したカードから位置が分かる | 高 |
| 端末メタデータ漏れ | EXIF の GPS がそのまま外部に出る | 高 |
| 説明不足 | ユーザーが位置情報利用を理解していない | 高 |

### 5-2. 必須の意思決定
| 論点 | 決めること | 推奨 |
|---|---|---|
| 位置情報の保存 | 保存する / しない / 条件付き | **条件付き保存** |
| 公開カードへの反映 | 位置名のみ / 地図付き / 非表示 | **デフォルトは位置名のみまたはぼかし** |
| 住所レベル表示 | する / しない | **しない** |
| 自宅近辺のぼかし | する / しない | **する** |
| 共有時のGPS削除 | する / しない | **する** |

### 5-3. 位置情報付き共有カードの危険性
SNS共有カードは魅力がある一方で危険です。

- 写真だけでなく **場所名・地図・ルート感** が出る
- 旅行中は良くても、**自宅周辺や通勤圏** を晒す恐れがある
- 家族・恋人共有では問題なくても、**外部SNS投稿で事故** が起きやすい

### 5-4. 推奨判断
- **共有カードは「外部公開用」と「限定共有用」を分ける**
- 外部公開用は **位置情報を弱める** か **選択制** にする
- GPSの精密座標は、原則として **共有カードにそのまま出さない**

## 6. AI利用リスク

### 6-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| データ過剰送信 | 写真内容、位置、文脈を送りすぎる | 高 |
| 機微情報流出 | 家族、恋人、子ども、場所情報が外部送信される | 高 |
| 誤生成 | 事実誤認のタイトルや本文を生成する | 中〜高 |
| コスト増 | 生成回数に比例して費用が増える | 高 |
| APIキー漏洩 | OpenAI APIキーをアプリに直書きすると危険 | 非常に高 |

### 6-2. 必ず決めること
| 項目 | 決めること |
|---|---|
| AIへ送るデータ範囲 | タイトル候補生成に何を送るか |
| 画像を送るか | 画像そのものを送信するか、メタデータのみか |
| 位置情報の粒度 | 緯度経度を送るか、地名だけにするか |
| 共有ノート情報 | 他メンバーの写真・文を送るか |
| 再生成回数 | 無制限か、上限ありか |

### 6-3. 推奨判断
- **OpenAI APIキーをモバイルアプリに入れない**
- AI は **Cloud Functions / Edge Functions / 自前バックエンド経由**
- **送信データは最小限**
- まずは **メタデータ中心の生成** を優先
- 画像内容解析は、必要性が明確になるまで抑制

---

## 7. SNS共有カードリスク

### 7-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 公開事故 | 意図せず個人情報や位置情報を含むカードを公開 | 高 |
| ブランド事故 | デザイン崩れでアプリの印象が悪くなる | 中 |
| 生成コスト | 画像レンダリングが重い | 中〜高 |
| 共有先差分 | Instagram / LINE / X / Facebook で見え方が違う | 中 |
| 権利問題 | 写真の著作権・肖像権・共同編集者の同意 | 高 |

### 7-2. 必ず決めること
| 論点 | 決めること |
|---|---|
| カードに載せる情報 | タイトル、日付、場所、本文、地図のどこまで |
| 比率 | 1:1 / 4:5 / 9:16 の優先順 |
| 位置表示 | ぼかす / エリア名のみ / ピン表示 |
| ブランド表記 | Made with Memory Note の表示可否 |
| 事前確認 | プレビューを必須にするか |

### 7-3. 推奨判断
- **プレビュー必須**
- **外部SNS向けは位置情報を抑える**
- **友人・家族の顔写真の扱いに注意**
- 公開前にユーザーに **「このカードは外部公開される」** と明示する

## 8. 外部API・コストリスク

### 8-1. 主なコストリスク
| サービス | コスト要因 | リスク |
|---|---|---|
| Firebase | 認証、DB、ストレージ、転送量、関数実行 | 利用増で増大 |
| 地図API | SDK、タイル、表示回数、ルート表示 | 予想以上に増える可能性 |
| Geocoding API | 逆ジオコーディング呼び出し回数 | 写真数に比例して増える |
| OpenAI API | 生成回数、文字量、画像解析 | AI機能利用増で増大 |
| クラウドストレージ | 画像容量、転送、バックアップ | 画像中心で急増しやすい |

### 8-2. 必須の意思決定
| 項目 | 決めること |
|---|---|
| クラウド基盤 | **決定済み: Firebase** |
| 地図SDK | MapKit / Google Maps SDK |
| Geocoding | 使う / 使わない / 限定利用 |
| AI使用量制限 | 1日上限、月上限、再生成回数 |
| 画像保存量 | 1ノートあたりの保存上限 |

### 8-3. 推奨判断
- **無料枠前提で設計**
- **地図とGeocodingはキャッシュ前提**
- AI は **回数制限あり**
- ストレージは **圧縮・サムネイル優先**
- まずは **コスト上限の見える化** が必要

## 9. ストア審査リスク

### 9-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 写真権限説明不足 | 何のために写真アクセスするか不明瞭 | 審査落ちの可能性 |
| 位置情報説明不足 | GPSをなぜ使うか不明瞭 | 審査落ちの可能性 |
| AI説明不足 | どのデータを外部送信するか不明瞭 | 審査・信頼リスク |
| 共有の誤解 | どこまで公開されるか分かりにくい | 審査・炎上リスク |
| データ削除導線不足 | アカウント削除やデータ削除が不十分 | 審査・法務リスク |

### 9-2. App Store / Google Play で必ず説明すべきこと
| 説明項目 | 必須内容 |
|---|---|
| 写真利用 | 選択写真をノート化するために使用 |
| 位置情報利用 | 撮影場所を地図に表示するために使用 |
| AI利用 | タイトル・短文日記生成のために送信される情報 |
| クラウド保存 | ノートと画像を複数端末で使うため |
| 共有範囲 | 共有ノートのメンバーに何が見えるか |
| 削除方法 | アカウント、ノート、画像の削除導線 |

### 9-3. 推奨判断
- **プライバシーポリシー、利用規約、AI説明を先に確定**
- ストア提出前に **「写真・位置・AI」説明文を画面実装**
- **許可を取る前提の UI** にする

## 10. 技術リスク

### 10-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 権限実装の複雑化 | 共同編集と共有が絡む | 高 |
| データ整合性 | 画像、DB、権限、削除の整合が難しい | 高 |
| オフライン復旧 | アップロード中断や失敗時の回復 | 中〜高 |
| モバイル差異 | iOS / Android の写真・権限挙動差 | 中〜高 |
| 地図実装 | SDK選定と表示コスト | 中 |
| AIバックエンド設計 | キー保護、レート制限、失敗時処理 | 高 |

### 10-2. 要検証事項
- iOS / Android で写真メタデータ取得が安定するか
- 選択写真のみで十分に UX が成立するか
- 共有ノートの権限変更が破綻しないか
- 削除が完全に反映されるか
- AI失敗時にフォールバックできるか

## 11. ビジネスリスク

### 11-1. 主要リスク
| リスク | 内容 | 影響 |
|---|---|---|
| 価値理解不足 | 写真管理との違いが伝わらない | 高 |
| 共有需要の過大評価 | 実際には個人利用が中心かもしれない | 中〜高 |
| コスト回収不明 | クラウド費用に見合う収益化が未確定 | 高 |
| 差別化不足 | Google Photos / Apple Photos / Day One に埋もれる | 高 |

### 11-2. 重要な仮説
- 写真を選ぶだけでノート化される価値が刺さるか
- 共有ノートが実際に使われるか
- SNS共有カードが新規流入に効くか
- 位置情報付き体験に不安より価値が勝つか

### 11-3. 推奨判断
- v1 の成功指標は **共有数より、作成完了率と再利用率**
- SNS共有は重要だが、**事故が多ければ逆効果**
- まずは **コア体験の継続率** を優先
