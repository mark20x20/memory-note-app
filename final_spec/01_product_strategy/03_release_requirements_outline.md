# Release Requirements Outline v2

## 1. 要件定義の目的
- Release v1 を「実際に公開・運用できる本番アプリ」として成立させるための要件を整理する
- 写真を起点に、地図付き思い出ノートを自動生成し、共有・共同編集・SNS共有まで含めた最小実用範囲を定義する
- MVPではなく Release v1 として、クラウド前提・アカウント前提・共有前提で要件を固定する
- 今後の詳細設計、UI/UX設計、技術設計、テスト設計の入力にする

---

## 2. 業務要件 / サービス要件
### 2.1 サービスの提供価値
- ユーザーは写真を選ぶだけで、撮影日時・位置情報をもとに思い出ノートを作成できる
- 思い出ノートは、時系列・地図・場所別グループ・短文日記で構成される
- 1人利用だけでなく、友人・家族・恋人との共有・共同編集ができる
- 作成した思い出は、SNS共有カードとして美しく外部共有できる

### 2.2 Release v1 の業務スコープ
- アカウント登録、ログイン、プロフィール管理
- 写真アップロードとクラウド保存
- EXIFベースの日時・GPS取得
- 自動整理、地図表示、AI下書き生成
- 保存、編集、削除、検索、振り返り
- 共有ノート、権限管理、メンバー管理
- SNS共有カード生成と端末共有
- プライバシー説明、利用規約、問い合わせ導線

### 2.3 Release v1 で扱わないこと
- 公開SNSフィード、いいね、コメント、フォロー
- 常時GPSトラッキング
- 旅行予約、口コミ、ナビゲーション
- Google Photos型の全面バックアップ
- フォトブックや動画生成の本格機能

---

## 3. ユーザー要件
### 3.1 想定ユーザー
- 旅行、カフェ巡り、グルメ、お出かけを記録したい人
- カップル、友人グループ、家族で思い出を共有したい人
- 日記は続かないが、写真はたくさん撮る人
- Instagram / LINE / X で見栄えよく共有したい人

### 3.2 ユーザーの期待
- 写真から自動で整理されること
- 場所や流れが見返しやすいこと
- 自分やメンバーの言葉で補足できること
- 共有してもプライバシーが守られること
- 削除や公開範囲が明確であること

### 3.3 ユーザーの主操作
- 写真選択
- ノート生成確認
- タイトル・本文・場所名編集
- 共有メンバー招待
- ノート閲覧・検索・編集・削除
- SNS共有カード出力

---

## 4. 機能要件

### 4.1 アカウント・認証要件
- メール、Apple、Google 等によるアカウント登録ができること
- ログイン、ログアウトができること
- ログイン状態を維持できること
- プロフィールとして表示名、アイコンを保持できること
- アカウント削除ができること
- 機種変更や複数端末利用を前提にした認証であること

### 4.2 クラウドDB要件
- ノート、メンバー、権限、タグ、メタ情報をクラウドDBに保存できること
- Owner / Editor / Viewer の権限情報を保持できること
- 削除・離脱・招待状態を履歴または状態として管理できること
- 複数端末から同じデータを参照できること
- データ整合性を担保できること

### 4.3 クラウドストレージ要件
- 写真をクラウドストレージへアップロードできること
- 元画像、圧縮画像、サムネイルを分けて扱えること
- ノート削除時に関連画像を削除できること
- 共有ノートの権限に応じて参照制御できること
- 保存コストを抑えるために圧縮保存を優先できること

### 4.4 写真選択・アップロード要件
- OS標準の写真選択UI、または同等の選択UIを提供すること
- 選択した写真のみを対象にアップロードできること
- 複数写真を一括選択できること
- アップロード進捗と失敗状態を表示できること
- アップロード失敗時に再試行できること

### 4.5 メタデータ取得要件
- 写真の撮影日時を取得できること
- EXIF の GPS 情報を取得できること
- 向き、ファイル情報など最低限のメタデータを扱えること
- GPSがない写真をエラー扱いにせず処理できること
- メタデータ欠損時の代替表示を行えること

### 4.6 自動整理要件
- 撮影日時順に写真を並べられること
- 近い時間・位置の写真を場所別にグループ化できること
- ノート単位で一日の流れを表現できること
- AIまたはルールベースでタイトル案を生成できること
- AIまたはルールベースで短文日記案を生成できること
- 保存前に整理結果をプレビューできること

### 4.7 地図表示要件
- 写真の位置を地図上にピン表示できること
- ノート詳細で地図を確認できること
- 場所別のまとまりが視覚的に分かること
- 場所名推定を表示できること
- 位置情報がない場合はその旨を分かる形で表示できること

### 4.8 AI生成要件
- AIタイトル生成ができること
- AI短文日記生成ができること
- AI生成結果をユーザーが手動修正できること
- AI失敗時はフォールバック文を返せること
- AI送信データを最小化できること
- OpenAI APIキーをモバイルアプリへ直接埋め込まないこと
- AI生成は Cloud Functions、Edge Functions、または自前バックエンド経由とすること

### 4.9 ノート保存・編集・削除要件
- ノートを保存できること
- 保存後にタイトル、本文、場所名、メモを編集できること
- ノート詳細を閲覧できること
- ノート一覧を表示できること
- ノート削除ができること
- 削除時に関連データの削除範囲を明示できること
- 参照不能な写真がある場合は状態を表示できること

### 4.10 共有ノート・共同編集要件
- 共有ノートを作成できること
- メンバーを招待できること
- Owner / Editor / Viewer を区別できること
- Member list を確認できること
- Editor は写真追加、メモ編集、場所名編集ができること
- Viewer は閲覧のみできること
- Owner は削除、権限変更、メンバー管理ができること
- メンバーが共有ノートから離脱できること
- メンバー削除時の扱いが明確であること
- 共有ノート削除時のデータ扱いが明確であること

### 4.11 SNS共有カード要件
- SNS共有カードを生成できること
- 1:1 正方形、4:5 フィード、9:16 ストーリーに対応できること
- メイン写真、タイトル、日付、場所名、短文、写真枚数、スポット数を含められること
- ミニマップまたはルート感を表現できること
- アプリ名、ロゴ、「Made with Memory Note」表記を含められること
- 出力前プレビューを表示できること
- 端末に保存できること
- OS共有シートで共有できること
- Instagram、LINE、X、Facebook で使いやすい導線を持てること

### 4.12 検索・タグ・カレンダー要件
- ノート一覧を日付順に表示できること
- カレンダー表示で振り返れること
- タグを付与できること
- タイトル、場所名、本文で簡易検索できること
- お気に入り登録ができること
- 月次・年次サマリーを表示できること
- On This Day を表示できること

### 4.13 設定・ヘルプ要件
- 権限説明画面を表示できること
- 写真、位置情報、AI、共有の使い道を説明できること
- プライバシーポリシーをアプリ内から確認できること
- 利用規約をアプリ内から確認できること
- 問い合わせ導線を設けること
- 基本設定画面を提供すること

---

## 5. 非機能要件
- iOS / Android で利用できること
- 日本語UIを初期対応とすること
- クラウド保存前提でも高速に動作すること
- 写真アップロード失敗時の復旧性を持つこと
- 権限がなくてもクラッシュしないこと
- 削除処理が不完全にならないこと
- 将来のクラウド基盤変更に耐えられる設計であること
- 高額外部API依存を避けること
- 監視、クラッシュレポート、基本分析を備えること

- ホーム / ノート一覧画面
- 写真選択画面
- ノート生成プレビュー画面
- ノート詳細画面
- ノート編集画面
- 地図表示画面
- カレンダー画面
- 共有ノートメンバー管理画面
- SNS共有カード生成画面
- 設定画面

## 8. 権限・プライバシー要件
- 選択写真のみアクセスを基本とすること
- 写真・位置情報・AI利用目的を明示すること
- 共有範囲を明示すること
- 自動公開しないこと
- データ削除導線を用意すること
- 位置情報は写真EXIF中心で扱い、常時追跡はしないこと
- 共有ノートの編集権限を明確にすること
- ユーザーが何を外部送信するか理解できること
- プライバシーポリシーと利用規約を公開前に確定すること

---

## 9. AI要件
- AI生成はバックエンド経由で実行すること
- モバイルアプリに OpenAI API キーを直接保存しないこと
- AIには必要最小限のデータだけ送信すること
- 生成対象はタイトル、短文日記、必要に応じた補助文に限定すること
- 生成失敗時は代替生成を返すこと
- ユーザーが生成結果を必ず編集できること
- AI利用の説明を明示すること

---

## 10. 地図・位置情報要件
- 写真EXIFのGPSを扱えること
- GPSがない写真も扱えること
- 地図ピン表示ができること
- 場所名推定ができること
- 近い位置・時間を同じスポットとしてまとめられること
- 常時位置情報取得はしないこと
- 位置情報の公開有無は明確に制御すること

---

## 11. 保存・同期要件
- クラウド保存を前提とすること
- データは複数端末で参照可能であること
- 保存途中の中断に耐えられること
- 削除内容がクラウドに反映されること
- 写真は元画像、圧縮画像、サムネイルを分けて管理すること
- 将来の同期競合に備えた構造にすること

---

## 12. 共有・共同編集要件
- 共有ノートの招待、参加、離脱ができること
- Owner / Editor / Viewer の責務を明確化すること
- 共有メンバーが写真追加できること
- メモや場所名を共同で編集できること
- 共有ノートの削除ルールを定義すること
- メンバー削除時の可視性・残存データを定義すること

---

## 13. SNS共有カード要件
- 画像フォーマットは 1:1 / 4:5 / 9:16 を扱うこと
- 共有カードの構成要素を固定化すること
- 生成結果を保存・共有できること
- ブランド要素を小さく自然に入れること
- 実際のSNS投稿時の見え方を崩さないこと

---

## 14. 管理・運用要件
- 監視、クラッシュログ、基本分析を取ること
- 不具合問い合わせを受けられること
- データ削除対応の運用ができること
- 権限トラブル時の復旧方針を持つこと
- 外部API障害時の代替動作を用意すること
- コスト増加が起きる箇所を把握できること

---

## 15. 受け入れ基準
- ユーザーが登録してログインできる
- 写真を選んでアップロードできる
- 撮影日時・GPSを取得して整理できる
- 地図付き思い出ノートを保存できる
- 編集後に再保存できる
- 共有メンバーを招待し、権限別に操作できる
- SNS共有カードを生成し、端末保存・共有できる
- 削除がデータ全体に反映される
- 権限・AI・位置情報の説明が確認できる

## 決定事項
- Release v1 はクラウド前提とする
- Release v1 はアカウント前提とする
- Release v1 は共有ノート・共同編集を含める
- Release v1 は Owner / Editor / Viewer を採用する
- Release v1 は SNS共有カードを必須とする

## 未決定事項
- Cloud provider の最終選定
- 画像保存ポリシーの原本保存範囲
- AIデータ送信の最終範囲
- 共有ノートの削除ルール詳細
- 位置情報の公開仕様
- 課金方針

## 不足している証拠
- 具体的な利用率、継続率、共有率の実測
- 日本市場での共有ニーズの定量検証
- AI生成文の満足度に関するユーザーテスト結果

## リスク
- 共有・権限設計が複雑で実装コストが上がる
- 写真と位置情報を扱うためプライバシー要求が高い
- SNS共有カードがコスト・工数を押し上げる
- 外部API依存が増えると運用コストが上がる
