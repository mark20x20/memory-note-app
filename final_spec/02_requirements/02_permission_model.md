# Permission Model v2

## 1. 目的

本モデルは、**思い出ノートアプリ**における以下を安全に成立させるための権限設計です。

- 共有ノート
- 共同編集
- 削除
- SNS共有カード

特に、Release v1 の前提である以下を満たすことを目的とします。

- **Owner / Editor / Viewer** を中心にした単純で実装可能なモデル
- **ノート削除は Owner のみ**
- **Viewer は閲覧のみ**
- **Cloud Firestore Security Rules に落とし込み可能**
- **プライバシー事故と誤削除を防ぐ**
- **共同編集を v1 で実用範囲に収める**

---

## 2. 権限モデルの基本方針

### 2.1 基本原則

| 原則 | 内容 |
|---|---|
| 最小権限 | それぞれのロールは必要最小限のみ許可する |
| 削除は慎重 | ノート削除は Owner のみに限定する |
| Viewer は安全側 | Viewer は閲覧のみ、編集・削除・招待変更は不可 |
| 共有と公開を分離 | 共有ノートとSNS外部共有は別権限で扱う |
| 位置情報は厳格 | 外部公開時は位置情報を弱める |
| Firestore で判定可能 | サーバ側でロール判定できるデータ構造にする |
| 例外を増やしすぎない | v1 は Owner / Editor / Viewer 以外のロールを増やさない |

### 2.2 権限設計の結論

| 項目 | 決定 |
|---|---|
| ロール体系 | Owner / Editor / Viewer |
| ノート削除 | Owner のみ |
| メンバー管理 | Owner のみ |
| 権限変更 | Owner のみ |
| 写真追加 | Owner / Editor |
| 写真削除 | Owner / Editor は可、ただし制限付き |
| 本文・場所名編集 | Owner / Editor |
| Viewer | 閲覧のみ |
| SNS共有カード生成 | Owner / Editor / Viewer いずれも可、ただし公開前プレビュー必須 |
| 外部投稿用保存 | 共有ノートの参加者は可、ただし位置情報ぼかし必須 |

## 3. ロール定義

### 3.1 Owner

| 項目 | 定義 |
|---|---|
| 役割 | ノートの所有者。最終責任者 |
| 基本思想 | 破壊的操作を行える唯一のロール |
| 代表例 | ノート作成者、管理者、関係者代表 |

#### Owner ができること
- ノート閲覧
- ノート編集
- ノート削除
- 写真追加
- 写真削除
- 場所名編集
- 本文編集
- AI生成実行
- AI再生成
- タグ編集
- メンバー招待
- メンバー削除
- 権限変更
- Owner移譲
- 共有カード生成
- 共有カード外部投稿用保存
- ノートから離脱
- アカウント削除時の扱いの決定

#### Owner の注意点
- Owner が退会する前に **必ず Owner 移譲** が必要
- Owner は削除事故の責任主体になる
- Owner 以外に最終権限を分散しない

---

### 3.2 Editor

| 項目 | 定義 |
|---|---|
| 役割 | 思い出ノートを編集できる共同編集者 |
| 基本思想 | 共同編集は可能だが、破壊的操作は制限する |
| 代表例 | 友人、恋人、家族メンバー |

#### Editor ができること
- ノート閲覧
- ノート編集
- 写真追加
- 写真削除
- 場所名編集
- 本文編集
- AI生成実行
- AI再生成
- タグ編集
- 共有カード生成
- 共有カード外部投稿用保存
- ノートから離脱

#### Editor ができないこと
- ノート削除
- メンバー招待
- メンバー削除
- 権限変更
- Owner移譲
- アカウント削除時の最終判断

#### Editor の注意点
- Editor に削除権限を広げると誤削除リスクが上がる
- v1 では **ノート削除不可** を厳守する
- 写真削除は「ノートから外す」意味に限定し、実データ削除はルール化する

---

### 3.3 Viewer

| 項目 | 定義 |
|---|---|
| 役割 | 閲覧専用メンバー |
| 基本思想 | 安全な共有範囲の最小ロール |
| 代表例 | 参加はするが編集しない人 |

#### Viewer ができること
- ノート閲覧
- 共有カード生成
- 共有カード外部投稿用保存
- ノートから離脱

#### Viewer ができないこと
- ノート編集
- ノート削除
- 写真追加
- 写真削除
- 場所名編集
- 本文編集
- AI生成実行
- AI再生成
- タグ編集
- メンバー招待
- メンバー削除
- 権限変更
- Owner移譲

#### Viewer の注意点
- Viewer は「見られるが壊せない」ことが重要
- Viewer は共同編集に参加しない前提でも成立する
- Viewer の存在で共有範囲を広くしつつ安全に保てる

## 5. 操作ごとの詳細ルール

### 5.1 ノート閲覧

| 項目 | ルール |
|---|---|
| 閲覧可能者 | Owner / Editor / Viewer |
| 閲覧範囲 | 自分が参加している共有ノートのみ |
| 非公開要素 | 削除済みデータ、権限外メタ情報 |
| Firestore | read 条件で member 判定を行う |

---

### 5.2 ノート編集

| 項目 | ルール |
|---|---|
| 編集可能者 | Owner / Editor |
| 編集対象 | タイトル、本文、場所名、タグ、写真並び順 |
| Viewer | 編集不可 |
| 注意 | 編集対象にメンバー管理情報は含めない |

---

### 5.3 ノート削除

| 項目 | ルール |
|---|---|
| 削除可能者 | Owner のみ |
| Editor | 不可 |
| Viewer | 不可 |
| 削除の意味 | ノート本体、写真参照、派生データの削除または無効化 |
| 推奨挙動 | ソフト削除 → 一定条件で物理削除 |

#### 推奨理由
- 誤削除は最も重大な事故
- 共同編集では「編集できる人」と「消せる人」を明確に分けるべき
- v1 は復元機構よりも事故防止を優先する

---

### 5.4 写真追加

| 項目 | ルール |
|---|---|
| 追加可能者 | Owner / Editor |
| Viewer | 不可 |
| 追加対象 | 選択写真のみ |
| 注意 | アプリが勝手に端末内全写真へアクセスしない |

---

### 5.5 写真削除

| 項目 | ルール |
|---|---|
| 削除可能者 | Owner / Editor |
| Viewer | 不可 |
| 削除の意味 | まずは「ノートから外す」を基本 |
| 実データ削除 | Firestore/Storage 上の画像削除を伴う |
| 推奨 | UI上は「写真を削除」だが、内部では削除範囲を明示する |

#### 推奨運用
| 種類 | 削除可否 |
|---|---|
| ノート内表示から外す | 可 |
| Storage の圧縮画像削除 | 条件付きで可 |
| サムネイル削除 | 条件付きで可 |
| 原本削除 | 原本を保存している場合のみ対象 |

---

### 5.6 場所名編集

| 項目 | ルール |
|---|---|
| 編集可能者 | Owner / Editor |
| Viewer | 不可 |
| 対象 | 地名、スポット名、通称 |
| 注意 | 住所レベルの露出は避ける |

---

### 5.7 本文編集

| 項目 | ルール |
|---|---|
| 編集可能者 | Owner / Editor |
| Viewer | 不可 |
| 対象 | 一言メモ、AI生成文の修正 |
| 注意 | 他メンバーの投稿を勝手に削らない運用が望ましい |

---

### 5.8 AI生成実行 / AI再生成

| 項目 | ルール |
|---|---|
| 実行可能者 | Owner / Editor |
| Viewer | 不可 |
| 対象 | タイトル、短文日記の生成 |
| 再生成 | 可。ただし回数制限推奨 |
| 注意 | AI送信データは最小化する |

---

### 5.9 タグ編集

| 項目 | ルール |
|---|---|
| 編集可能者 | Owner / Editor |
| Viewer | 不可 |
| 対象 | 旅行、カフェ、デート、家族など |
| 注意 | タグの乱用を避けるため、v1 は最小限でよい |

---

### 5.10 メンバー招待

| 項目 | ルール |
|---|---|
| 招待可能者 | Owner のみ |
| Editor | 不可 |
| Viewer | 不可 |
| 理由 | 共有範囲の拡大は責任が大きい |

---

### 5.11 メンバー削除

| 項目 | ルール |
|---|---|
| 削除可能者 | Owner のみ |
| Editor | 不可 |
| Viewer | 不可 |
| 注意 | 退出と削除は別概念 |

---

### 5.12 権限変更

| 項目 | ルール |
|---|---|
| 変更可能者 | Owner のみ |
| Editor | 不可 |
| Viewer | 不可 |
| 注意 | Viewer→Editor 変更などは Owner が明示的に行う |

---

### 5.13 Owner移譲

| 項目 | ルール |
|---|---|
| 実行可能者 | Owner のみ |
| 対象 | 共有ノート内の他メンバー |
| 推奨条件 | 移譲先が Editor 以上であること |
| 注意 | Owner 不在状態を作らない |

---

### 5.14 共有カード生成

| 項目 | ルール |
|---|---|
| 生成可能者 | Owner / Editor / Viewer |
| 保存可能者 | Owner / Editor / Viewer |
| 外部投稿用保存 | 可 |
| 注意 | 共有カードは「閲覧権限」とは独立して生成できる |

---

### 5.15 ノートから離脱

| 項目 | ルール |
|---|---|
| 離脱可能者 | Owner / Editor / Viewer |
| 離脱の意味 | 自分自身を共有ノートから外す |
| 注意 | 他人を外す操作ではない |
| Owner 離脱 | 事前に Owner 移譲必須 |

---

### 5.16 アカウント削除時の扱い

| 項目 | ルール |
|---|---|
| 自分のアカウント削除 | 可能 |
| 影響範囲 | 自分が Owner のノート、参加中ノート、招待状態 |
| Owner の場合 | 先に Owner 移譲を要求 |
| Editor / Viewer | 自分の参加状態を解除 |
| 注意 | 他人のノートを勝手に消さない |

## 6. 招待モデル

### 6.1 招待方式

| 方式 | 採用 | 用途 |
|---|---|---|
| 招待リンク | 採用 | 共有ノート参加 |
| メール招待 | 採用 | 相手を特定して招待 |
| QR招待 | 任意 | 将来拡張候補 |

### 6.2 招待の基本ルール

| 項目 | ルール |
|---|---|
| 招待発行者 | Owner のみ |
| 招待先ロール | 発行時に Owner が指定 |
| 招待期限 | 必須 |
| 招待取り消し | 必須 |
| 再発行 | 可 |
| 参加条件 | 招待受諾 + 認証済み |

### 6.3 招待リンク漏洩対策

| 対策 | 内容 |
|---|---|
| 期限付き | 一定時間または一定日数で失効 |
| 1回限り | 参加後は無効化できる |
| ロール固定 | 招待時に Editor/Viewer を固定 |
| 失効再生成 | Owner が再発行可能 |
| トークン難読化 | 推測されにくいランダム文字列を利用 |
| 参加者確認 | 参加前に共有ノート名を見せる |

### 6.4 招待取り消し

| 項目 | ルール |
|---|---|
| 取り消し可能者 | Owner のみ |
| 取り消し対象 | 未参加の招待 |
| 効果 | トークン無効化 |
| 注意 | 参加済みメンバーの権限変更とは別 |

## 7. 削除ルール

### 7.1 Ownerによるノート削除

| 項目 | ルール |
|---|---|
| 削除可能者 | Owner のみ |
| 対象 | ノート本体、画像参照、メタデータ、共有情報 |
| 推奨 | ソフト削除後に物理削除 |
| 確認 | 二段階確認必須 |

### 7.2 Editorによる写真削除可否

| 項目 | ルール |
|---|---|
| Editor の削除 | 可 |
| ただし | ノート全体削除は不可 |
| 推奨 | 写真単位の削除は「ノートから外す」中心 |
| 注意 | 共同編集の利便性と誤削除防止のバランスを取る |

### 7.3 Viewerの削除不可

| 項目 | ルール |
|---|---|
| Viewer の削除権限 | なし |
| 理由 | 誤操作防止、権限分離 |
| 実装 | Firestore Rules でも拒否する |

### 7.4 メンバー離脱時の写真・メモの扱い

| 項目 | ルール |
|---|---|
| 離脱前の投稿 | 原則ノートに残す |
| 離脱後の見え方 | 参加者としては外す |
| 投稿者表示 | 可能なら「離脱済みメンバー」として残す |
| 注意 | 他人の投稿を勝手に消さない |

### 7.5 Owner退会時のOwner移譲

| 項目 | ルール |
|---|---|
| Owner退会 | 直接不可 |
| 必須条件 | 先に Owner移譲 |
| 代替 | 最終手段としてノート削除（Ownerのみ） |
| 注意 | Owner不在を禁止する |

### 7.6 アカウント削除時の共有ノート扱い

| 状況 | ルール |
|---|---|
| Owner が削除 | 先に移譲必須 |
| Editor が削除 | 自分の参加解除 |
| Viewer が削除 | 自分の参加解除 |
| ノート内データ | 共同所有データは原則保持 |
| 個人プロフィール | 自分のアカウントと紐づく情報は削除対象 |

## 8. 共同編集ルール

### 8.1 同時編集の初期方針

| 項目 | ルール |
|---|---|
| v1 の方針 | 簡易上書き防止 |
| 編集方式 | 楽観的ロックに近い形を推奨 |
| 競合発生時 | 最終更新前に警告 |
| 高度な差分統合 | v1では必須にしない |

### 8.2 最終更新者

| 項目 | ルール |
|---|---|
| 表示 | 「最終更新者」「最終更新時刻」を保存 |
| 用途 | 競合判断、追跡、安心感 |
| 推奨 | ノート単位と主要要素単位で保持 |

### 8.3 競合時の簡易対策

| 対策 | 内容 |
|---|---|
| 保存前確認 | 他人が更新していないか確認 |
| 更新時刻比較 | 古い版で上書きしない |
| 軽い警告 | 「他のメンバーが更新しました」表示 |
| 再読み込み導線 | 最新版を取得して再編集 |

### 8.4 編集履歴をv1で保存するか

| 項目 | 方針 |
|---|---|
| 詳細履歴 | v1では不要 |
| 最低限履歴 | 最終更新者、更新日時は保存 |
| 差分履歴 | Phase 2 以降で検討 |

### 8.5 v1.1以降に回す高度な履歴管理

| 機能 | 位置づけ |
|---|---|
| 編集履歴一覧 | v1.1以降 |
| 差分表示 | v1.1以降 |
| 復元機能 | Phase 2 以降 |
| 衝突自動解決 | Phase 3 以降 |

## 10. Firestore Security Rules設計方針

### 10.1 基本方針

| 方針 | 内容 |
|---|---|
| 認可はサーバ側で判定 | クライアント任せにしない |
| membership を参照 | ノートごとのメンバー情報で判定する |
| role を明示保存 | Owner / Editor / Viewer を member document に持つ |
| ソフト削除対応 | deletedAt / status を持つ |
| 書き込み範囲を限定 | フィールド単位の更新制御を検討する |

### 10.2 推奨データ構造

| コレクション | 役割 |
|---|---|
| users | ユーザー情報 |
| notes | ノート本体 |
| notes/{noteId}/members | メンバーとロール |
| notes/{noteId}/photos | 写真メタ情報 |
| notes/{noteId}/invites | 招待トークン |
| notes/{noteId}/activity | 最終更新・軽い履歴 |

### 10.3 Rules で見たい条件

| 条件 | 内容 |
|---|---|
| 自分がメンバーか | notes/{noteId}/members に uid があるか |
| ロールは何か | member.role を確認 |
| Owner か | role == "owner" |
| Editor 以上か | role in ["owner", "editor"] |
| Viewer か | role == "viewer" |
| 削除済みか | note.status != deleted |

### 10.4 Firestore Rules の設計メモ

| 操作 | 判定例 |
|---|---|
| read note | member なら可 |
| update note | owner/editor なら可 |
| delete note | owner のみ |
| add member | owner のみ |
| update member role | owner のみ |
| leave self | 自分自身の member doc のみ可 |
| photo add | owner/editor のみ |
| photo delete | owner/editor のみ、もしくは owner のみの厳格版 |

### 10.5 推奨実装方針

| 項目 | 推奨 |
|---|---|
| 権限判定 | Firestore Rules + バックエンド併用 |
| 破壊的操作 | Cloud Functions 経由も検討 |
| 招待検証 | Cloud Functions でトークン検証 |
| 削除 | 直接 delete ではなく soft delete 推奨 |

## 11. Human Approval が必要な権限判断

| 論点 | なぜ承認が必要か |
|---|---|
| Editor に写真削除を許すか | 誤削除事故に直結する |
| Editor にメンバー招待を許すか | 共有範囲が広がる |
| Viewer に共有カード保存を許すか | 外部公開事故の入口になる |
| Owner 退会時の自動移譲条件 | 所有権事故に直結する |
| アカウント削除時の共有ノート残存可否 | 法務・UX・信頼に影響する |
| 削除後の復元期間 | 誤削除対策と削除完了のバランス |
| 外部公開カードの位置情報粒度 | プライバシー事故に直結する |

## 主要リスク

| リスク | 内容 |
|---|---|
| 誤削除 | Editor に削除権限を広げすぎる |
| 所有権不在 | Owner 不在で管理不能になる |
| 権限漏れ | Firestore Rules の実装ミスで閲覧・編集が漏れる |
| 位置情報事故 | SNS共有カードで精密位置が漏れる |
| 競合上書き | 共同編集で変更が潰れる |

必要であれば次に、**この権限モデルをそのまま Firestore の collections / documents / rules 条件に落とした実装仕様**まで書けます。
