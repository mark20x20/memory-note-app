# AI Data Policy v2

## 1. 目的

本ポリシーは、Memory Note App / 思い出ノートアプリにおける **OpenAI API利用時のデータ送信範囲** と **Cloud Functions 経由の実行方針** を定めるものです。

目的は以下です。

- 写真から思い出ノートを自動生成できるようにする
- 共有ノート・共同編集・SNS共有カードの体験を損なわずにAIを使う
- 写真・位置情報・個人情報の漏えいリスクを抑える
- **OpenAI APIキーをモバイルアプリに直接入れない**
- AI処理を **Cloud Functions 経由** に統一し、監査・制御・課金管理をしやすくする
- コストを抑えつつ、v1で実用的なAI体験を成立させる

---

## 2. 基本方針

### 2.1 中核原則

| 原則 | 方針 |
|---|---|
| 最小送信 | OpenAIへ送るデータは、生成に必要な最小限に限定する |
| 画像原則非送信 | **写真本体は原則送らない** |
| 位置情報慎重 | **精密GPS座標は原則送らない** |
| キー保護 | OpenAI APIキーは **モバイルアプリに入れない** |
| サーバ経由 | AI生成は **Cloud Functions / Edge Functions / 自前バックエンド経由** |
| 共同編集配慮 | 他メンバーの情報は、本人同意や明示要件がない限り送らない |
| 編集可能 | AI生成結果は必ずユーザーが修正できるようにする |
| 失敗時継続 | AI失敗でもノート作成フローは止めない |
| 監査可能 | 送信内容・回数・失敗はサーバ側で追跡できるようにする |

### 2.2 データ設計上の前提

- 画像保存は **圧縮画像 + サムネイル中心**
- AIは原則として **メタデータ駆動** で使う
- AIは「自動生成の補助」であり、記録の主導権はユーザーにある
- 生成文は事実の断定ではなく、**ドラフト** として扱う
- 外部公開向けのカード生成とは分離する

### 2.3 v1でのAI位置づけ

v1でAIを使うのは以下に限定します。

- タイトル生成
- 短文日記生成
- 要約生成
- フォールバック文生成

それ以外は、原則として Phase 2 以降に回します。

---

## 3. AI利用範囲

### 3.1 扱う機能

| 機能 | 使い方 |
|---|---|
| タイトル生成 | 日付、場所名、スポット情報から思い出ノートのタイトル案を作る |
| 短文日記生成 | ノートの流れを1〜数文で説明する下書きを作る |
| 要約生成 | ノート全体の短いサマリーを作る |
| フォールバック文生成 | AI失敗時にルールベースで最低限の文を返す |
| 補助整形 | ユーザーメモの要約、タグの補助提案 |
| v1でやらないこと | 画像内容の本格認識、人物特定、感情推定、公開SNS文面の自動投稿、旅行計画生成 |

### 3.2 v1でやらないこと

以下は **本ポリシーの対象外** または **v1では非対象** です。

- 写真そのものを解析して詳細な被写体認識を行う
- 人物の特定、顔認識、性別推定、年齢推定を行う
- 常時位置履歴をAIへ送る
- 他メンバーのメッセージを無条件に外部送信する
- 旅行予約、口コミ、SNS投稿文の自動生成を行う
- 公開フィード向けの炎上リスクのある文章を自動生成する

## 4. AIへ送るデータ

以下は、**送信可否** を明確に分けた表です。

| データ | 送信可否 | 理由 | 注意点 | 例 |
|---|---|---|---|---|
| ノートタイトル候補 | 条件付きで送信可 | 既存タイトルの改善や再生成に利用するため | そのまま個人情報を含めない | `東京旅行`, `鎌倉のおでかけ` |
| 日付 | 可 | 生成の基本軸になるため | 日時は日単位・時間帯単位に丸める | `2026-06-07` |
| 時間帯 | 可 | 朝・昼・夕方などの表現に有用 | 分単位は不要なことが多い | `昼`, `夕方` |
| 場所名 | 可 | タイトル・短文生成の中心情報 | 住所ではなく地名・施設名程度にする | `渋谷`, `鎌倉駅前` |
| エリア名 | 可 | 精密座標を避けつつ文脈を伝えられる | 広すぎると精度低下 | `東京都`, `横浜エリア` |
| 写真枚数 | 可 | ボリューム感の表現に使える | 画像実体ではなく数値のみ | `12枚` |
| スポット数 | 可 | 一日の流れや巡回感を表現できる | 1地点ごとの詳細は不要 | `3スポット` |
| スポット順序 | 可 | 時系列の流れを文章化しやすい | 詳細ルートは不要 | `カフェ → 神社 → ランチ` |
| ユーザーメモ | 条件付きで可 | ユーザーの意図を反映できる | 余計な個人情報を含めないよう整形 | `また行きたい`, `雨だったけど楽しかった` |
| タグ | 可 | 文体や分類の補助になる | タグ数は少数に絞る | `travel`, `cafe`, `family` |
| 共有メンバー情報 | 原則条件付き | 共同体験の文脈には有用 | 個人名や連絡先は送らない。役割のみ推奨 | `友人`, `家族`, `恋人` |
| 写真本体 | 原則不可 | プライバシー・コスト・漏えいリスクが高い | 例外時のみ明示同意が必要 | 送らない |
| サムネイル | 原則不可、必要時のみ | 視覚補助になる場合がある | 画像送信が必要な機能は最小限に限定 | 通常は送らない |
| EXIF | 原則不可、必要項目のみ可 | 不要な個人情報が含まれやすい | 完全EXIFは送らない | 撮影日時のみ抽出して送る |
| GPS座標 | 原則不可 | 住所特定・生活圏露出の恐れ | 精密座標は送らない。地名・エリア名へ丸める | `35.6,139.7` は送らない |
| AI過去生成結果 | 条件付きで可 | 再生成・文体変更に使える | 長文を全部送らず要点のみ | 直前のタイトル案 |
| 位置情報の丸め結果 | 可 | 精密座標の代替として使える | 住所レベルは避ける | `渋谷区`, `東京都` |
| ノートの集約値 | 可 | 生成品質向上に有用 | 冗長でも最小限に | `総写真数`, `総スポット数` |

### 4.1 推奨する送信セット

#### タイトル生成用
- ノートの日付
- 時間帯
- 場所名
- エリア名
- スポット数
- 写真枚数
- タグ
- ユーザーメモの要約
- 既存タイトル案があればそれ

#### 短文日記生成用
- ノートの日付
- 時間帯
- 場所名
- スポット順序
- スポット数
- 写真枚数
- ユーザーメモ
- タグ
- 必要に応じた過去生成結果の短縮版

#### 要約生成用
- ノートタイトル
- 日付
- 場所名
- 写真枚数
- スポット数
- 短い本文メモ
- 編集済み本文の要点

## 5. AIへ送らないデータ

以下は、**原則として送らない** データです。

| データ | 理由 |
|---|---|
| 写真本体 | 個人情報・背景情報・第三者情報を含み得るため |
| 精密GPS座標 | 自宅、勤務先、行動圏の特定につながるため |
| EXIF完全データ | 端末情報や位置情報が含まれうるため |
| 端末情報 | 個人識別や不要な追跡につながるため |
| メールアドレス | AI生成に不要で、識別情報として強すぎるため |
| 認証情報 | 絶対に送らない。漏えい時の被害が大きいため |
| 他メンバーの個人情報 | 同意なく外部送信できないため |
| 招待トークン | セキュリティ上の秘密情報だから |
| 住所レベルの位置情報 | 生活圏や自宅の特定につながるため |
| 電話番号 | 生成に不要で機微性が高いため |
| クレジットカード等の決済情報 | そもそも送信対象外 |
| 端末内の全写真一覧 | 選択写真だけに限定すべきため |
| ローカルファイルパス | 生成に不要で実装詳細を漏らすため |

### 5.1 特に厳禁なもの

- OpenAI APIキー
- Firebase Admin SDKの秘密鍵
- 認証トークン
- 共有リンクの秘密パラメータ
- 個人の連絡先一覧
- 顔写真そのものを含む未同意データ

## 6. Cloud Functions実行方針

### 6.1 基本構成

AI処理は必ず以下の経路で行います。

```text
Mobile App
  → Firebase Authで認証
  → Firestoreへリクエスト作成
  → Cloud Functions呼び出し
  → OpenAI APIへ送信
  → 結果をCloud Functionsで整形
  → Firestoreへ保存
  → Mobile Appへ返却
```

### 6.2 実行主体の役割

| コンポーネント | 役割 |
|---|---|
| Mobile App | ユーザー操作、生成依頼、結果表示 |
| Firebase Auth | 誰が実行したかの認証 |
| Firestore | 入力スナップショット、結果、制限状態の保存 |
| Cloud Functions | 送信前整形、API呼び出し、エラー制御、監査記録 |
| OpenAI API | テキスト生成 |
| Usage Limits | 回数制限、ノート単位制御、日次制御 |
| Error Handling | タイムアウト、失敗、リトライ、フォールバック |

### 6.3 実行ルール

1. **モバイルアプリからOpenAIへ直接通信しない**
2. **OpenAI APIキーはCloud Functionsの環境変数にのみ置く**
3. **Firestoreの権限確認後に実行する**
4. **1回の呼び出しで送るデータを最小化する**
5. **入力不足時は実行前に弾く**
6. **回数制限を超えたら拒否する**
7. **失敗時はフォールバック結果を返す**
8. **ログに機微情報を残さない**

### 6.4 Cloud Functionsで行うこと

- 認証確認
- 権限確認
- 入力整形
- 個人情報の除去
- 位置情報の丸め
- プロンプト生成
- OpenAI API呼び出し
- レスポンス整形
- Firestore保存
- 使用回数の更新
- エラーログ保存
- 監査ログ保存

### 6.5 Cloud Functionsで行わないこと

- 原本画像の恒久保存
- 認証情報の保持
- 個人情報の無加工送信
- 無制限な再生成
- クライアント代替のUIロジック
- 長時間の重いバッチ処理を1リクエスト内で完結させること

- noteId
- requestType
- language
- date
- timeOfDay
- placeName
- areaName
- photoCount
- spotCount
- spotOrder
- tags
- userMemoSummary
- existingTitle
- previousAiResultSummary

## 8. AI生成結果の保存方針

### 8.1 保存対象

AI結果は、`ai_results` に保存します。

| 項目 | 保存可否 | 説明 |
|---|---|---|
| inputSnapshot | 保存する | 何を送ったかの監査用 |
| outputText | 保存する | 生成結果本体 |
| outputCandidates | 必要時のみ保存 | 候補案がある場合 |
| promptVersion | 保存する | 再現性確保 |
| model | 保存する | 利用モデル追跡 |
| status | 保存する | success / failed / queued |
| errorMessage | 制限付きで保存 | 機微情報を含めない |
| userEdited | 保存する | ユーザー修正の有無 |
| editedText | 保存する | 修正後の最終文 |
| createdAt | 保存する | 監査用 |
| createdBy | 保存する | 実行者追跡 |

### 8.2 保存方針

- `inputSnapshot` は **最小限** にする
- 写真本体や精密GPSは `inputSnapshot` に含めない
- 個人情報をマスクした状態で保存する
- 保存期間は運用方針で定める
- 削除要求時は関連AI結果も削除対象にする

### 8.3 userEdited の扱い

| 項目 | 方針 |
|---|---|
| userEdited | ユーザーが修正したら true |
| editedText | 最終保存された文を保持 |
| outputText | 元のAI生成文として残す |
| 表示 | 必要に応じて「AI下書き」「編集済み」を区別 |

### 8.4 保存期間

推奨は以下です。

- Release v1 では、ノートが生きている間は保持
- ノート削除時に関連AI結果も削除
- アカウント削除時は関連AI結果も削除
- 監査目的で残す場合でも、個人情報は削除または匿名化する

## 9. 回数制限・コスト制御

### 9.1 基本方針

AIはコストが発生するため、使用上限を設けます。

### 9.2 制限項目

| 項目 | 例 | 方針 |
|---|---|---|
| 1日上限 | ユーザーごとの生成回数 | 設ける |
| 1ノート上限 | ノートごとの生成回数 | 設ける |
| 再生成上限 | 同一リクエストの連続再実行 | 設ける |
| 総文字量 | 入力・出力の上限 | 設ける |
| 同時実行数 | 同時に走るAIジョブ | 抑制する |

### 9.3 usage_limits の役割

`usage_limits` は、以下を管理します。

- 日次の生成回数
- ノート単位の生成回数
- 再生成回数
- レート制限
- 異常利用の検知
- プラン差分が将来入る場合の基盤

### 9.4 Abuse prevention

| 対策 | 内容 |
|---|---|
| 認証必須 | 未認証ユーザーは利用不可 |
| 1ユーザー単位制限 | 個別に上限管理 |
| 1ノート単位制限 | 同じノートの連打防止 |
| 短時間連続制限 | 連打・スパム防止 |
| 監査ログ | 異常利用を追えるようにする |
| エラー時増分禁止 | 失敗時に不必要な回数消費を避ける |

## 10. 失敗時フォールバック

### 10.1 失敗ケース

| 失敗要因 | 例 |
|---|---|
| AI失敗 | OpenAI側のエラー |
| APIタイムアウト | 応答が遅すぎる |
| rate limit | 呼び出し回数制限に達する |
| 入力不足 | 場所名や日付が足りない |
| 権限不足 | そのユーザーが実行できない |
| ネットワーク障害 | Functions や外部APIに接続できない |

### 10.2 フォールバック方針

- タイトルはルールベースで返す
- 日記は短い定型文で返す
- 要約は写真枚数と場所名から最低限生成する
- 失敗してもノート保存は継続する
- 再試行可能なUIを出す

### 10.3 ルールベース生成例

#### タイトル
- `鎌倉のおでかけ`
- `東京旅行 1日目`
- `家族で公園へ`

#### 短文日記
- `今日は写真を見返すのが楽しい一日でした。`
- `○○で過ごした時間を、写真と一緒に残しました。`
- `海沿いの景色とカフェをゆっくり楽しみました。`

#### 要約
- `鎌倉で海沿いを散歩し、カフェや神社を巡った思い出をまとめたノートです。`

### 10.4 フォールバックの位置づけ

フォールバックは「AIの代替」ではなく、**UXを止めないための安全弁**です。

## 11. ユーザー説明・同意

### 11.1 初回利用で説明すること

以下は、初回またはAI初使用時に説明します。

- AIで何をするか
- 何をAIに送るか
- 何をAIに送らないか
- 写真本体は原則送らないこと
- 精密GPS座標は原則送らないこと
- 生成文は編集できること
- 失敗時は自動で簡易文が出ること

### 11.2 設定画面で説明すること

設定画面には以下を置きます。

- AI利用のオン/オフ、または利用確認
- 送信データの説明
- 再生成回数制限の説明
- プライバシー説明へのリンク
- データ削除説明へのリンク

### 11.3 プライバシーポリシー文言の要点

記載すべき要点は以下です。

- AI機能のために、日時・場所名・写真枚数・スポット数・メモの一部を送ること
- 写真本体は原則送らないこと
- 精密な位置情報は送らないこと
- 送信先がOpenAIであること
- 生成結果は保存されることがあること
- 生成結果はユーザーが編集可能であること

## 12. プライバシー・セキュリティ

### 12.1 APIキー保護

**最重要事項: OpenAI APIキーをモバイルアプリに直接入れないこと。**

- クライアントアプリ埋め込みは禁止
- リバースエンジニアリングで漏えいしやすい
- Cloud Functions の環境変数または Secret Manager に格納する
- ログにキーを出さない
- エラーレスポンスにも含めない

### 12.2 最小データ送信

- 必要最小限だけ送る
- 個人名・連絡先・認証情報を除外する
- 住所レベルの位置情報を送らない
- 他メンバーの非公開情報を含めない

### 12.3 位置情報の丸め

推奨する丸め方は以下です。

| 元データ | 送信時の変換 |
|---|---|
| 緯度経度 | 地名、駅名、エリア名へ変換 |
| 住所 | 区・市レベルに丸める |
| 精密スポット | 施設名または一般名に変換 |

### 12.4 監査ログ

Cloud Functions 側で以下を記録します。

- 実行者 userId
- noteId
- requestType
- 実行日時
- 成功/失敗
- 使用トークン量の概算
- エラー種別

ただし、以下は記録しないか、強くマスクします。

- 生のプロンプト全文に含まれる個人情報
- APIキー
- 認証トークン
- 精密GPS
- 写真本体の内容

## 13. 受け入れ基準

以下を満たすことを受け入れ基準とします。

- モバイルアプリからOpenAIへ直接通信していない
- OpenAI APIキーがアプリ内に存在しない
- Cloud Functions 経由でAI生成ができる
- AI送信データが最小限に制限されている
- 写真本体が原則送信されない
- 精密GPS座標が原則送信されない
- タイトル、日記、要約、フォールバックが生成できる
- AI失敗時にノート作成が止まらない
- 生成結果が保存され、ユーザーが編集できる
- 回数制限と監査ログが機能する
- エラーログに機微情報が残らない

## 決定事項

- **OpenAI APIキーをモバイルアプリに直接入れない**
- AI生成は **Cloud Functions 経由** にする
- AIには **写真本体を原則送らない**
- AIには **精密GPS座標を原則送らない**
- AI送信データは **日時、場所名、写真枚数、スポット数、ユーザーメモ中心** にする
- AI結果は保存しつつ、ユーザーが編集できるようにする
- 回数制限と監査ログを必須にする

## 未決定事項

- 画像送信を許可する例外条件
- 共有メンバー情報の送信粒度
- 具体的な回数制限値
- 保存期間と削除条件
- OpenAIの利用モデルとプロンプト運用方法

## 不足している証拠

- AI生成文に対する日本ユーザーの満足度データ
- どの程度の位置情報粒度が許容されるかの実測
- 日記生成における「写真本体なし」での品質検証
- 共有メンバー情報を含めた場合のプライバシー受容性

## リスク

- 送信データが増えると、プライバシー事故と説明負荷が増える
- Cloud Functions に処理が集中すると、遅延とコストが増える
- 回数制限が弱いと、APIコストが膨らむ
- フォールバックが弱いと、AI失敗時にノート作成体験が崩れる

必要であれば次に、**Cloud Functions の API 仕様書（request/response schema 付き）** まで具体化します。
