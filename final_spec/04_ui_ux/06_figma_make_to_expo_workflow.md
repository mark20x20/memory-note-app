# Figma Make to Expo Workflow v2

## 1. 目的

本ドキュメントは、**Memory Note App / 思い出ノートアプリ** のUIを、  
**Figma Make で草案作成 → Figma Design で確認 → Codex で React Native + Expo + TypeScript に再実装** するための、実務向けワークフローを定義する。

このワークフローの目的は以下である。

- UI草案を高速に作る
- デザインの意図を Figma 上で確認する
- Figma Make の出力をそのまま本番採用せず、実装品質を担保する
- React Native + Expo + TypeScript 前提で再実装しやすくする
- 既存の Firebase バックエンド方針と整合させる
- 1画面ずつ確実に実装し、修正履歴を残す

---

## 2. 基本方針

### 2.1 技術前提

本ワークフローの前提は以下とする。

| 項目 | 方針 |
|---|---|
| モバイルフロントエンド | React Native + Expo + TypeScript |
| ルーティング | Expo Router |
| バックエンド | 既存 Firebase を継続利用 |
| 認証 | Firebase Authentication |
| DB | Cloud Firestore |
| 画像保存 | Firebase Storage |
| サーバ処理 | Cloud Functions |
| 権限制御 | Firestore Security Rules / Storage Rules |
| AI | OpenAI API を Cloud Functions 経由で利用 |
| 管理画面 | Streamlit + Firebase Admin SDK |
| ビルド | EAS Build |

### 2.2 ワークフローの考え方

- **Figma Make は草案生成用**
  - 画面レイアウト、構造、文言のたたき台を作るために使う
- **Figma Design は確認用**
  - レイアウト、余白、文言、導線、状態表現を確認するために使う
- **Codex は再実装支援用**
  - Figma の草案を元に、React Native + Expo で実装するための参考として使う
- **Figma Make のコードは本番コードにしない**
  - そのまま採用せず、必ず再実装する
- **Firebase バックエンド方針は維持する**
  - Auth / Firestore / Storage / Functions / Rules は既存方針に従う

### 2.3 実装順の原則

実装は次の順序で進める。

1. 画面目的を定義する
2. Figma Make に渡す md を作る
3. Figma Make で草案を生成する
4. Figma Design で確認する
5. UI 修正点を記録する
6. Codex へ実装情報を渡す
7. React Native + Expo で 1画面ずつ再実装する
8. iPhone 実機で確認する
9. 修正ログを残す

## 3. Figma Makeに渡すmd

### 3.1 役割

Figma Make に渡す md は、**画面の意図を伝えるための設計入力**である。  
コードを直接作らせる目的ではなく、次の情報を明示する。

- 画面名
- 画面の目的
- ユーザーが行う操作
- 主要UI要素
- 表示状態
- 遷移先
- エラー時の扱い
- 権限差分
- 日本語文言のたたき台

### 3.2 mdに含めるべき項目

各画面ごとに、少なくとも以下を含める。

| 項目 | 内容 |
|---|---|
| 画面ID | 例: `SCR-HOME-001` |
| 画面名 | 例: ホーム / ノート一覧 |
| 目的 | この画面で何を達成するか |
| 利用条件 | ログイン済み、Ownerのみなど |
| 主なUI | ボタン、リスト、カード、入力欄など |
| 状態 | 初期、空、読み込み中、エラー、権限なし |
| 主操作 | ユーザーが実行する操作 |
| 遷移 | 前画面 / 次画面 |
| 表示文言 | ボタン、見出し、補助文 |
| 注意点 | 位置情報、共有、削除などの注意 |

### 3.3 Figma Make用mdの書き方例

```md
# 画面ID
SCR-HOME-001

# 画面名
ホーム / ノート一覧

# 目的
ログイン済みユーザーが、自分の思い出ノート一覧を確認し、新規作成や検索に進めるようにする。

# 利用条件
- ログイン済み
- Firestoreからノート一覧を取得済み

# 主なUI
- ヘッダー
- 新規作成ボタン
- ノート一覧カード
- 検索ボタン
- 設定ボタン

# 表示状態
- 読み込み中
- 空状態
- 一覧あり
- エラー
- 権限なし

# 主操作
- ノートを開く
- 新規作成する
- 検索する
- 設定を開く

# 注意点
- 外部公開フィードは作らない
- SNSのいいねやコメントは不要
- 共有ノートも一覧に含める
```

### 3.4 Figma Makeに渡す範囲

Figma Make に渡すのは、**画面単位または画面群単位**とする。  
一度に大きすぎる範囲を渡さない。

推奨単位:

- オンボーディング一式
- 認証一式
- ホーム一式
- ノート詳細一式
- 共有カード一式
- 設定一式

## 4. Figma Make出力の保存方法

### 4.1 保存方針

Figma Make の出力は、**そのまま本番用にせず、参照用アーカイブとして保存**する。  
保存先は以下を基本とする。

```text
generated_ui/
  figma_make/
    2026-06-10/
      SCR-HOME-001/
      SCR-AUTH-001/
      SCR-NOTE-001/
```

### 4.2 保存対象

保存するものは以下。

- Figma Make の生成結果
- 画面キャプチャ
- 生成時の md
- 修正メモ
- レビュー指摘
- 再生成履歴

### 4.3 保存ルール

| 項目 | ルール |
|---|---|
| ディレクトリ名 | 日付 + 画面ID |
| ファイル命名 | `screen-id_description.ext` |
| 履歴保存 | 生成版と修正版を分ける |
| 本番コード分離 | Figma Make 出力をアプリ本体に混ぜない |
| 再利用 | 画像・文言の参考としてのみ利用 |

### 4.4 推奨構成

```text
generated_ui/
  figma_make/
    2026-06-10/
      SCR-HOME-001/
        prompt.md
        output.png
        notes.md
      SCR-NOTE-001/
        prompt.md
        output.png
        notes.md
```

## 5. Codexに渡す情報

### 5.1 目的

Codex には、**Figma Make の結果をそのまま貼るのではなく、実装に必要な情報を整理して渡す**。

### 5.2 渡すべき情報

| 項目 | 内容 |
|---|---|
| 画面ID | 実装対象の識別子 |
| 画面の目的 | 何を達成するか |
| 参考画像 | Figma Make / Figma Design の確認結果 |
| UI構成 | ヘッダー、カード、ボタン、フォームなど |
| 状態一覧 | loading / empty / error / success |
| 入出力 | どのデータを表示・編集するか |
| Firebase参照先 | Firestore / Storage / Functions |
| 権限 | Owner / Editor / Viewer の差分 |
| 遷移先 | Expo Router の遷移先 |
| 禁止事項 | 実装してはいけない挙動 |

### 5.3 Codexへの渡し方の例

```md
# 実装対象
SCR-NOTE-001 ノート詳細

# 目的
共有ノートの内容を閲覧し、写真、地図、メモ、メンバーを確認できる画面を実装する。

# 参考
- Figma Make 出力
- Figma Design 確認済みデザイン

# UI構成
- ヘッダー
- カバー写真
- タイトル
- 日付
- 地図プレビュー
- 写真一覧
- メモ本文
- メンバー一覧
- 共有カードボタン
- 編集ボタン
- 削除ボタン（Ownerのみ）

# 状態
- 読み込み中
- データあり
- データなし
- 権限なし
- 削除済み

# Firebase
- Firestore: memory_notes/{noteId}
- Firestore: memory_notes/{noteId}/members/{userId}
- Firestore: memory_notes/{noteId}/photos/{photoId}
- Functions: generateShareCard

# 権限
- Owner: 編集、削除、招待可能
- Editor: 編集可能、削除不可
- Viewer: 閲覧のみ

# 禁止事項
- OpenAI APIキーをアプリに入れない
- 共有ノートを公開URLにしない
- UIで隠すだけの権限制御にしない
```

## 6. UI再実装ルール

### 6.1 再実装の前提

Figma Make の出力は参考であり、**React Native + Expo + TypeScript で再構築する**。  
以下を守る。

- コンポーネントは再実装する
- データ取得は Firebase SDK を使う
- 状態管理はアプリ側の方針に合わせる
- 画面遷移は Expo Router で管理する
- モバイルの操作感に合わせて調整する

### 6.2 再実装時のルール

| 項目 | ルール |
|---|---|
| レイアウト | Figma をそのままコピーせず、RNに最適化する |
| 文言 | 日本語で自然に整える |
| 余白 | iPhone / Android 両方で見やすくする |
| 状態 | loading / empty / error を必ず実装する |
| 権限 | 画面表示だけでなくロジックでも制御する |
| 画像 | 端末サイズに応じて最適化する |
| 地図 | 高負荷にならない表示を優先する |
| 共有 | OS共有シート中心で実装する |
| AI | Functions 経由のみで呼ぶ |
| セキュリティ | クライアント直結の危険処理を避ける |

### 6.3 コンポーネント分割の考え方

- 1画面を小さな部品に分ける
- 再利用できる共通UIは `components/` に寄せる
- 権限付きUIは共通の guard を通す
- ノートカード、メンバー行、共有カードプレビューなどは共通化する

### 6.4 Expo Router の使い方

画面遷移は Expo Router で管理する。

例:

```text
app/
  (auth)/
  (main)/
  notes/
  share/
  settings/
```

ルール:

- 認証前後でルートを分ける
- 画面IDとルートを対応づける
- 深いリンクや招待導線を考慮する

## 7. 1画面ずつ実装する流れ

### 7.1 基本手順

1. 画面IDを1つ決める
2. Figma Make 用 md を作る
3. Figma Make で草案を出す
4. Figma Design で確認する
5. UI修正点を整理する
6. Codex に実装指示を渡す
7. React Native + Expo で画面実装する
8. iPhone 実機で確認する
9. 修正ログを残す
10. 次の画面へ進む

### 7.2 1画面実装時の確認項目

| 確認項目 | 内容 |
|---|---|
| レイアウト | 画面崩れがないか |
| 文言 | 日本語が自然か |
| 状態 | loading / empty / error があるか |
| 遷移 | 正しい画面へ進むか |
| 権限 | Viewer で編集が出ないか |
| データ | Firestore/Storage 参照が正しいか |
| 共有 | OS共有シートが起動するか |
| 実機 | iPhoneで操作しやすいか |

### 7.3 実装順の推奨

優先順は以下。

1. 認証
2. ホーム
3. 作成開始
4. 写真選択
5. アップロード進捗
6. 生成プレビュー
7. ノート詳細
8. 共有メンバー管理
9. 共有カード
10. 設定・プライバシー

## 8. iPhone実機確認

### 8.1 目的

iPhone 実機確認は、**Expo Go だけでは分からない本番近い挙動を確認するため**に行う。

確認したい項目:

- レイアウト崩れ
- Safe Area
- スクロール挙動
- 共有シート
- 写真選択
- 権限ダイアログ
- 地図表示
- 画像保存
- フォントサイズ
- ダークモード
- タップ領域

### 8.2 実機確認の進め方

1. EAS Build で実機確認用ビルドを作成する
2. TestFlight または開発用ビルドで iPhone に入れる
3. 主要画面を順に確認する
4. 問題点をスクリーンショット付きで記録する
5. 画面ごとに修正する

### 8.3 実機確認で優先する画面

- ログイン
- ホーム
- 写真選択
- アップロード進捗
- ノート詳細
- 地図
- 共有カード
- 設定

## 9. UI修正ログ

### 9.1 目的

UI修正ログは、**Figma Make → Figma Design → Codex 実装** の間で、どこをどう変えたかを追跡するために使う。

### 9.2 記録項目

| 項目 | 内容 |
|---|---|
| 日付 | 修正日 |
| 画面ID | 対象画面 |
| 修正前 | 旧状態 |
| 修正後 | 新状態 |
| 理由 | なぜ修正したか |
| 指示元 | Figma Make / Design / 実機確認 / レビュー |
| 実装者 | 変更担当 |
| 確認結果 | 反映済み / 再確認中 |

### 9.3 ログ例

```md
- 日付: 2026-06-10
- 画面ID: SCR-NOTE-001
- 修正前: 共有ボタンがヘッダー内に埋もれていた
- 修正後: 下部固定ボタンに変更
- 理由: iPhoneで押しやすくするため
- 指示元: 実機確認
- 実装者: Codex
- 確認結果: 反映済み
```

### 9.4 保存先の考え方

UI修正ログも、画面別に整理する。

```text
generated_ui/
  figma_make/
  ui_revision_log/
```

## 10. 禁止事項

以下は禁止とする。

### 10.1 技術面の禁止事項

- Flutter 前提を残す
- Supabase 前提を入れる
- OpenAI APIキーをモバイルアプリに埋め込む
- Firebase の既存方針を勝手に変える
- Firestore Security Rules を回避する
- Storage Rules を無視する
- 共有ノートを公開URLで丸出しにする
- いいね、コメント、フォローをアプリに入れる
- Figma Make の出力を本番コードとして直接使う
- Cloud Functions を通さず AI を直接呼ぶ
- 管理画面をモバイル側に混ぜる

### 10.2 運用面の禁止事項

- UI確認なしで実装を確定する
- 1画面に複数の大きな変更を同時に入れる
- 修正履歴を残さない
- 実機確認を省略する
- 権限差分を見落とす
- 位置情報の扱いを曖昧にする
- 削除導線を後回しにする

### 10.3 ディレクトリ運用の禁止事項

- Figma Make 出力を本番ソースと同じ場所に置く
- 保存先を統一しない
- 日付や画面IDなしで管理する
