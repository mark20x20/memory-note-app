# Operations Overview v2

## 1. 目的

Memory Note App / 思い出ノートアプリ の Release v1 以降における、**運用・保守・監視・改善・障害対応・課金運用・サポート運用**の全体像を定義します。

本運用設計の目的は以下です。

- 本番公開後も安定してサービスを継続運用できるようにする
- 写真、位置情報、共有、AI、課金を安全に扱う
- 障害やコスト増加を早期検知する
- AIを使って運用負荷を下げつつ、最終判断は人間が行う
- 変更管理を明確にし、事故を防ぐ
- コード変更なしで、価格・制限・機能フラグを調整できるようにする
- App Store / Google Play の課金整合性を保つ
- ユーザー対応を属人化させず、再現性のある運用にする

---

## 2. 運用保守の基本思想

### 2.1 基本原則

| 原則 | 内容 |
|---|---|
| AI補助、人間承認 | AIは分析・提案・下書きまで。最終決定は人間 |
| 事故防止優先 | 誤削除、誤課金、誤公開を最優先で防ぐ |
| 最小権限 | 管理者権限は最小人数・最小範囲 |
| 変更は追跡可能 | すべての運用変更を監査ログに残す |
| 価格と機能は分離管理 | コード変更なしで制御できるが、課金整合は必ず確認 |
| 安全側デフォルト | 迷ったら停止、非公開、制限強めにする |
| 低コスト継続 | 無駄なAPI利用、Storage増加、AI再生成を抑える |
| プライバシー最優先 | 位置情報・画像・個人情報は最小限のみ扱う |
| 監視は多層化 | Analytics / Crashlytics / Logs / Firestore audit logs を併用 |

### 2.2 運用の前提

- Firebase を中心に運用する
- AI は OpenAI API を Backend 経由で使う
- 管理画面は **Streamlit** を採用してよい
- 運用設定は Firestore の `app_config` を中心に管理する
- ただし、**本番デプロイ、課金仕様変更、Security Rules変更、データ削除、ユーザー制裁、プライバシーポリシー変更は Human Approval 必須**
- AI が本番コードを直接変更したり、本番へ勝手に反映したりしない
- ストア課金は App Store Connect / Play Console と整合する必要がある

## 3. AI運用保守の位置づけ

### 3.1 AIの役割

AIは以下を担います。

- 監視データの要約
- 障害兆候の分析
- コスト増加の検知
- 日次・週次レポート作成
- 問い合わせの一次分類
- 定型返信ドラフト作成
- 変更提案の作成
- 運用手順書の下書き
- ルール逸脱の検知補助
- 機能フラグや制限値の変更案提示

### 3.2 AIの位置づけ

AIは **運用担当者の代替ではなく、運用担当者の補助** です。

- AIは「見る・まとめる・提案する」
- 人間は「判断する・承認する・実行する」

この分離を崩さないことが重要です。

### 3.3 AIの利用対象

- Firebase Analytics の異常傾向
- Crashlytics のクラッシュ集中
- Cloud Functions の失敗率
- Firestore / Storage のコスト兆候
- OpenAI API の使用量・失敗率
- Map / Geocoding API の利用量
- サブスク継続率や解約兆候
- サポート問い合わせの分類

## 4. 人間運用の位置づけ

### 4.1 人間が担うべき役割

人間は以下を担います。

- 本番運用方針の決定
- Release v1 の運用基準の維持
- 価格・プラン・機能範囲の決定
- 重大障害時の停止判断
- ユーザー制裁やアカウント対応
- データ削除の最終承認
- ストア提出・審査対応
- プライバシーポリシー / 利用規約の更新判断
- Security Rules の変更判断
- 本番デプロイの実行判断

### 4.2 人間運用の原則

- AIの提案は参考情報
- 重要変更は必ず人間が内容確認
- 変更前に影響範囲を可視化
- 変更後に監査ログを残す
- 緊急時の判断基準を事前に決める

## 6. AIが自動化してはいけない範囲

AIが**自律実行してはいけない**のは、重大リスクを伴う操作です。

### 6.1 禁止事項
- 本番コードの自動変更
- 本番デプロイ
- 課金仕様の自動変更
- ストア課金商品の変更
- Security Rules の自動変更
- ユーザーデータの削除
- ユーザー制裁の自動実行
- プライバシーポリシーの自動変更
- 利用規約の自動変更
- 公開範囲の自動拡大
- 監査なしの設定反映

### 6.2 理由
- 誤操作の影響が大きい
- 巻き戻しが困難
- 法務・審査・課金整合に影響する
- プライバシー事故につながる
- ストア審査に抵触する可能性がある

---

## 7. Human Approvalが必要な操作

以下は必ず人間承認が必要です。

| 操作 | 理由 |
|---|---|
| 本番デプロイ | サービス全体への影響が大きい |
| 課金仕様変更 | App Store / Play Console と整合が必要 |
| サブスク価格変更 | 売上・審査・返金影響がある |
| プラン範囲変更 | 既存ユーザーへの影響が大きい |
| AI回数制限の大幅変更 | コストとUXに直結 |
| ストレージ上限の変更 | コストと既存データに影響 |
| Feature Flag の本番反映 | 体験を即時変える |
| Security Rules変更 | アクセス制御事故の可能性 |
| データ削除 | 復元不可リスクが高い |
| ユーザー制裁 | 誤判定リスクと法的配慮が必要 |
| プライバシーポリシー変更 | 法務・ストア審査に影響 |
| 利用規約変更 | 同意条件が変わる |
| 緊急停止解除 | 再開判断が重要 |
| 外部API課金プランの変更 | コストと継続性に影響 |

## 11. Streamlit Admin Dashboard方針

Streamlit を管理画面として採用してよいです。  
ただし、**簡単に作れることと、安全に使えることを両立**させます。

### 11.1 Streamlit採用理由
- 迅速に作れる
- 内部管理画面に向いている
- 表示と簡易操作がしやすい
- AIレポートとの相性が良い
- コストが低い

### 11.2 Streamlitで行うこと
- ダッシュボード表示
- KPI確認
- 変更案の編集
- 承認ワークフロー
- feature flag の切替
- `app_config` 更新
- コスト監視
- 問い合わせ一覧の確認
- AIレポートの確認
- 緊急停止ボタン表示

### 11.3 Streamlitで行わないこと
- 本番コードの直接編集
- データ削除の即時実行
- 承認なしの課金変更
- Security Rules の直接反映
- ユーザー制裁の自動実行
- 秘密情報の平文表示

### 11.4 権限設計
- Admin Viewer
- Admin Operator
- Approver
- Super Admin

### 11.5 セキュリティ要件
- 管理画面は認証必須
- IP制限または追加認証を推奨
- 操作ログを全件保存
- 重要操作は二段階確認
- 本番環境と検証環境を分離
