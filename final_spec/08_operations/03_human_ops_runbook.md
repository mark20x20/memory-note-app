# Human Ops Runbook v2

## 1. 目的

この手順書は、**Memory Note App / 思い出ノートアプリ** の本番運用を、人間が安全かつ継続的に管理するためのものです。

対象は以下です。

- Firebase / OpenAI / Map API / Store 課金 / Storage / Cloud Functions の監視
- 価格・プラン・機能フラグ・AI回数制限・ストレージ上限の調整
- 障害対応、問い合わせ対応、権限トラブル対応
- アカウント削除、データ削除、課金変更などの承認フロー
- AI Ops が作成したレポートや提案の確認・判断

この運用では、**AIは補助** であり、**最終判断と危険操作は人間** が行います。

### 運用の基本原則
- 迷ったら止める
- 破壊的操作は必ず承認する
- 変更は必ず監査ログを残す
- 価格と課金整合はストア側を優先する
- 個人データは最小限だけ扱う
- 本番コード変更、本番デプロイ、データ削除は自動化しない

## 2. 管理者の役割

### 2.1 役割区分

| 役割 | 主な責務 | 実行可否 |
|---|---|---|
| 運用担当 | 日次監視、問い合わせ一次対応、設定変更案の確認 | 一部可 |
| 承認者 | 価格変更、緊急停止解除、データ削除承認 | 必須 |
| 技術担当 | Firebase/Functions/Storage/Rules の調整 | 一部可 |
| 法務・責任者 | プライバシー、規約、制裁、課金方針の承認 | 必須 |
| サポート担当 | 問い合わせ整理、返信ドラフト確認 | 可 |

### 2.2 権限設計の前提
- 管理画面は **Streamlit** でよい
- 管理画面の操作は **管理者認証必須**
- 変更対象は `app_config` や管理用コレクション
- ただし以下は **Human Approval 必須**:
  - 本番デプロイ
  - 課金仕様変更
  - Security Rules 変更
  - データ削除
  - ユーザー制裁
  - プライバシーポリシー変更

### 2.3 管理者が毎日見るもの
- 異常アラート
- Crashlytics
- Cloud Functions エラー
- OpenAI 利用量
- Firebase / Storage コスト
- 課金売上と解約
- 問い合わせ件数
- 緊急停止フラグ

## 3. 日次運用

### 3.1 朝の確認
毎営業日、まず以下を確認します。

#### 確認順
1. **AI Ops 日次レポート**
2. **重大アラート**
3. **Crashlytics**
4. **Cloud Functions エラー**
5. **コスト推移**
6. **問い合わせ**
7. **課金異常**
8. **緊急停止状態**

#### 見る場所
- Streamlit Admin Dashboard
- Firebase Console
- Cloud Functions logs
- Crashlytics
- Firestore audit logs
- OpenAI usage dashboard
- App Store Connect / Play Console
- サポート管理表

### 3.2 日次で判断すること
- 前日比で急増している指標があるか
- 特定機能だけ失敗していないか
- 課金失敗や返金が増えていないか
- AI回数が想定を超えていないか
- Storage が想定より増えていないか
- ユーザーから同じ不具合報告が来ていないか

### 3.3 日次の実施タスク
- AIレポートの確認
- 重大アラートの既読・分類
- 問い合わせの一次分類
- 低リスクな設定確認
- 変更案の承認待ちキュー確認
- 監査ログの確認

### 3.4 日次で変更してよい範囲
- 低リスクの文言修正案の作成
- 機能フラグの**提案**
- AI回数制限の**提案**
- サポート返信ドラフトの確認
- 監視しきい値の確認

### 3.5 日次では変更しないもの
- 本番デプロイ
- 課金仕様変更
- Rules 変更
- データ削除
- プラン価格変更
- ユーザー制裁
- 緊急停止解除

## 8. 障害発生時の初動

### 8.1 初動の基本
障害を見つけたら、まず **被害拡大を止める** ことを優先します。

### 8.2 初動手順
1. 障害内容を記録
2. 影響範囲を確認
3. 緊急停止が必要か判断
4. 直近変更を確認
5. ログを確認
6. ユーザー影響を分類
7. 一時回避策を決定
8. 必要なら外部案内を出す

### 8.3 優先順位
- P0: データ消失、課金事故、広範囲クラッシュ、権限漏れ
- P1: 一部機能障害、AI失敗、アップロード失敗増加
- P2: 軽微なUX不具合、表示崩れ

### 8.4 すぐ止めるべき例
- 共有ノートが他人に見える
- 権限が破られている
- 課金が二重計上している疑い
- データ削除が誤って広がる
- APIコストが急増している

## 9. コスト増加時の対応

### 9.1 監視対象
- Firebase Auth
- Firestore
- Storage
- Cloud Functions
- OpenAI API
- Map API
- Store 課金関連コスト

### 9.2 まず見るもの
- 何が増えたか
- いつから増えたか
- どの機能が原因か
- どのユーザー層か
- 変更直後かどうか

### 9.3 対応順
1. 原因候補を特定
2. AI Ops コストレポートを確認
3. 無駄な再生成や再試行がないか確認
4. キャッシュ不足がないか確認
5. フラグで止められるか確認
6. 必要なら機能を一時停止
7. 必要なら制限値を下げる

### 9.4 変更候補
- AI回数制限を下げる
- 共有カード再生成を抑える
- 写真保存サイズを見直す
- 地図呼び出しのキャッシュ強化
- 一時的に高コスト機能をOFFにする

### 9.5 変更に承認が必要なもの
- 本番の `app_config` を大きく変えること
- 価格・プラン変更
- 課金仕様変更
- ストレージ上限の恒久変更

## 10. OpenAI API障害時の対応

### 10.1 兆候
- AIタイトル生成が失敗
- AI日記生成が失敗
- OpenAI timeout が増加
- 生成が遅い
- 429 / 5xx が増える

### 10.2 対応方針
- AIが止まってもノート作成は継続する
- フォールバック文を使う
- 再試行回数を制限する
- 必要ならAI機能を一時停止する

### 10.3 対応手順
1. Functions ログを確認
2. OpenAI 側障害情報を確認
3. usage_limits とコストを確認
4. 429 なら回数制限を見直す
5. timeout ならプロンプト短縮を検討
6. 必要なら AI 機能フラグを OFF
7. 復旧後に再開

### 10.4 使ってよい代替
- ルールベースのタイトル
- 定型文の短文日記
- ユーザー手入力

## 11. Firebase障害時の対応

### 11.1 対象
- Auth
- Firestore
- Storage
- Functions
- Analytics
- Crashlytics

### 11.2 対応の考え方
Firebase は基盤なので、障害時は **代替運用を最小化** し、復旧待ちを基本とします。

### 11.3 判断
- Auth 障害: ログインできないため新規操作を止める
- Firestore 障害: 保存・読込に影響。書き込み停止を検討
- Storage 障害: 写真アップロード停止を検討
- Functions 障害: AIや削除処理を停止
- Analytics/Crashlytics 障害: 監視は弱くなるが、ユーザー機能は継続

### 11.4 対応手順
1. Firebase 状況を確認
2. 影響機能を特定
3. 必要な機能だけ止める
4. ユーザー向け案内を準備
5. 復旧後に再有効化

## 12. Storage容量増加時の対応

### 12.1 見るもの
- 圧縮画像の増加量
- サムネイルの増加量
- 共有カードの残存数
- 原本保存が混ざっていないか
- 削除漏れがないか

### 12.2 対応順
1. Storage 増加元を特定
2. 原本保存が有効になっていないか確認
3. 削除漏れジョブを確認
4. キャッシュや一時ファイル残留を確認
5. 共有カードの保持期間を確認
6. 必要なら自動削除の設定を強める

### 12.3 変更候補
- 共有カードの保存期限短縮
- 一時ファイルの即時削除
- サムネイルサイズ見直し
- 原本保存の無効化維持
- 画像圧縮率の調整

## 13. Cloud Functionsエラー増加時の対応

### 13.1 見るもの
- どの function が落ちているか
- timeout か logic error か
- 依存API失敗か
- 権限エラーか
- 再試行で改善するか

### 13.2 対応
1. 関数名を特定
2. エラー分類
3. 入力異常を確認
4. 外部API障害を確認
5. 直近デプロイを確認
6. 必要なら該当機能を停止
7. 修正案を作る

### 13.3 代表例
- `generateDiary` だけ失敗 → AI側か入力整形
- `generateShareCard` だけ失敗 → 画像生成ロジック
- `deletePhoto` 失敗 → Storage 削除権限や整合性
- `createInvitation` 失敗 → 権限や招待コレクション

## 14. ユーザー問い合わせ対応

### 14.1 受付の基本
問い合わせはカテゴリ分けして扱います。

#### カテゴリ
- ログインできない
- 写真がアップロードできない
- AIが動かない
- 共有ノートが見えない
- 権限が違う
- 課金・サブスク
- 削除依頼
- 位置情報
- 画面不具合
- その他

### 14.2 対応手順
1. 受信
2. カテゴリ分類
3. 緊急度判定
4. 必要なら追加情報依頼
5. 返信ドラフト確認
6. 対応完了
7. 監査記録

### 14.3 返信前に確認すること
- 本人確認が必要か
- データ削除依頼か
- 課金関連か
- 共有ノートの権限問題か
- プライバシー事故の可能性があるか

### 14.4 AI利用
AIには以下を任せてよいです。
- 問い合わせの要約
- 返信ドラフト
- 分類候補
- FAQ候補

AIに任せてはいけないこと:
- データ削除判断
- 制裁判断
- 課金返金判断
- 規約解釈の最終決定

## 15. アカウント削除・データ削除対応

### 15.1 原則
削除は**最も危険な操作**の一つです。  
必ず人間が確認します。

### 15.2 対応区分
- アカウント削除
- ノート削除
- 写真削除
- 共有ノートからの離脱
- データ保持期間満了による削除

### 15.3 実施前確認
- 本人依頼か
- 本人確認済みか
- 削除範囲は明確か
- 共有ノートへの影響はあるか
- Owner なら移譲済みか
- 復元不可であることを説明したか

### 15.4 実施手順
1. 依頼内容を記録
2. 対象を確認
3. 影響範囲を確認
4. 承認を取得
5. 削除ジョブを実行
6. 完了を確認
7. 監査ログを残す

### 15.5 禁止
- 問い合わせだけで即削除
- AIの判断だけで削除
- まとめて大量削除
- 削除範囲を曖昧にした実行

---

## 16. 共有ノート・権限トラブル対応

### 16.1 よくあるトラブル
- ノートが見えない
- 編集できない
- 招待が届かない
- Viewer なのに編集できるように見える
- Owner がいなくなった
- 誤ってメンバーを外した

### 16.2 確認順
1. 当該ノートの membership
2. role
3. status
4. 招待状態
5. Owner の有無
6. Rules / Functions のエラー
7. UI の表示不具合かどうか

### 16.3 対応
- Viewer なのに編集できる表示 → UI不具合として修正
- 実際に権限が壊れている → 緊急対応
- Owner 不在 → Owner移譲を実施
- 招待失敗 → 再発行
- 誤削除 → 復元可否を確認し、無理なら説明

### 16.4 重要
権限トラブルは **見た目の問題** と **実権限の問題** を分けて確認します。

## 17. サブスク / プラン設定変更時の対応

### 17.1 変更対象
- 月額価格
- 年額価格
- 無料プラン範囲
- AI回数上限
- Storage 上限
- 機能フラグ
- 特典内容

### 17.2 必須確認
- App Store Connect と Play Console の商品ID
- ストア価格との整合
- 既存ユーザーへの影響
- 課金中ユーザーの扱い
- 返金や再購入への影響
- 利用規約・ヘルプの整合

### 17.3 手順
1. 変更案をAIに作らせる
2. 影響分析を見る
3. ストア側設定を確認
4. 管理画面の `app_config` 更新案を作成
5. Human Approval を取得
6. 本番反映
7. 反映後に監査ログを確認

### 17.4 禁止
- ストア設定と違う価格をアプリに出す
- 片側だけ変更する
- 承認なしで課金仕様を変える

## 18. Human Approval手順

### 18.1 承認が必要な操作
- 本番デプロイ
- 課金仕様変更
- Security Rules 変更
- データ削除
- ユーザー制裁
- プライバシーポリシー変更
- 緊急停止解除
- 価格変更
- プラン範囲変更

### 18.2 承認フロー
1. AIまたは担当者が変更案を作成
2. 影響範囲を記載
3. ロールバック案を記載
4. 承認者に提示
5. 承認/差戻し
6. 承認ログを保存
7. 実行
8. 実行結果を記録

### 18.3 承認時に必ず見る項目
- 変更内容
- 影響範囲
- 失敗時の戻し方
- 課金整合
- 法務影響
- ユーザー影響
- 実施者
- 承認者
- 時刻

### 18.4 承認なしで実行してはいけない例
- `app_config` で価格を変える
- Feature Flag で新機能を本番ON
- 個別ユーザーを制裁
- データ削除を実行
- Rules を変えてアクセス範囲を広げる

## 19. ロールバック手順

### 19.1 ロールバックが必要な条件
- リリース後にクラッシュ増加
- 課金事故
- 共有事故
- 権限事故
- コスト急増
- アップロード失敗増加

### 19.2 手順
1. 影響範囲を止める
2. 最新変更を特定
3. 変更前の設定を確認
4. 元設定に戻す
5. 監査ログを残す
6. 復旧確認
7. 再発防止を記録

### 19.3 ロールバックの対象
- Feature Flag
- `app_config`
- 生成設定
- UI文言
- サーバ側パラメータ
- ストア公開設定の運用手順

### 19.4 ロールバックできないもの
- 既に削除したデータ
- 既に反映されたストア審査結果
- 既存課金の完全取り消し
- 外部SNSに投稿された内容

## 21. AI Opsレポートの読み方

### 21.1 日次レポート
見るべきポイント:
- 昨日より急に悪化した指標
- 新しいエラー
- コスト増加
- 問い合わせ増加
- 共有や課金の異常

### 21.2 週次レポート
見るべきポイント:
- トレンド
- 再発パターン
- 低使用機能
- コスト最適化候補
- 価格・プラン見直し候補

### 21.3 月次レポート
見るべきポイント:
- 売上とコスト
- 解約率
- 継続率
- 機能別採算
- 問い合わせ傾向
- ストアレビューの変化

### 21.4 レポートの使い方
- そのまま実行しない
- まず根拠を見る
- 変更前後比較を見る
- 承認が必要か確認
- 反映後の監視計画を決める

## 22. 禁止事項

### 22.1 絶対禁止
- AIが本番コードを勝手に変更する
- AIが本番デプロイする
- AIが課金仕様を勝手に変える
- AIがデータ削除を勝手に実行する
- AIがユーザー制裁を勝手に実行する
- AIがプライバシーポリシーを勝手に変更する

### 22.2 運用上の禁止
- 口頭だけで重要変更
- 監査ログなしの変更
- 承認者不在の変更
- 課金整合が取れていない変更
- 削除範囲が曖昧な実行
- 個人データをAIへ無制限送信
- 緊急停止を解除したまま監視しないこと

### 22.3 設計上の禁止
- 管理画面から本番コードを直接編集
- 管理画面から Security Rules を無承認で変更
- 個別ユーザーの秘密情報を表示
- 精密位置情報を外部共有用にそのまま出す
- ストア価格と異なる表示をする
