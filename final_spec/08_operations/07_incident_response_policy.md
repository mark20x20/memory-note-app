# Incident Response Policy v2

## 1. 目的

本ポリシーは、Memory Note App / 思い出ノートアプリ の本番運用において、障害・品質劣化・コスト異常・セキュリティ事故・課金不整合を安全かつ迅速に扱うための基準を定めます。

本アプリは、写真を起点に思い出ノートを自動生成し、共有ノート・共同編集・SNS共有カードを提供するため、障害の影響が以下に広がりやすい前提で運用します。

- 写真アップロードが止まる
- ノート保存が失敗する
- 共同編集の権限が壊れる
- 位置情報や共有情報が漏れる
- AI生成が失敗または暴走する
- ストア課金やサブスクの整合が崩れる

本ポリシーの原則は以下です。

- 早期検知する
- 被害拡大を止める
- AI Ops は分析と提案までに留める
- 人間が最終判断する
- 危険操作は必ず承認を取る
- 変更、削除、停止は監査ログを残す

---

## 2. Incident分類

以下をインシデント対象とします。

### 2.1 Authentication outage
認証機能の障害です。

#### 例
- メールログインできない
- Apple / Google ログインが失敗する
- セッション復元に失敗する
- ログアウト後に再ログインできない

#### 影響
- 新規利用不能
- 既存ユーザーの利用停止
- 課金導線にも影響する可能性あり

---

### 2.2 Firestore outage
Firestore の読み書き障害です。

#### 例
- ノート一覧が表示されない
- 保存できない
- メンバー情報が読めない
- 権限判定が失敗する

#### 影響
- ノート作成・閲覧・共同編集が停止
- データ不整合のリスクが高い

---

### 2.3 Storage upload failure
画像アップロード障害です。

#### 例
- 写真が送れない
- 圧縮画像が保存されない
- サムネイル生成後にアップロード失敗する
- 共有カード画像が保存できない

#### 影響
- 思い出ノートの作成体験が壊れる
- ストレージコストや再試行の暴走につながる

---

### 2.4 Cloud Functions failure
Cloud Functions の障害です。

#### 例
- AI生成関数が落ちる
- 招待関数が失敗する
- 削除連鎖が失敗する
- 共有カード生成が失敗する

#### 影響
- AI、招待、削除、共有処理が停止
- 連鎖的にユーザー体験が悪化する

---

### 2.5 OpenAI API failure
OpenAI API の障害または劣化です。

#### 例
- タイトル生成が失敗
- 日記生成がタイムアウト
- 429 で再生成できない
- 生成品質が著しく低下

#### 影響
- AI補助機能が使えない
- 再試行増加でコストが増える

---

### 2.6 Map API failure
地図・ジオコーディング API の障害です。

#### 例
- 地図が表示されない
- 場所名が推定できない
- ピンが出ない
- 共有カードのミニマップ生成に失敗する

#### 影響
- 地図付きノートの価値低下
- 共有カードの見栄え低下

---

### 2.7 Cost spike
コスト異常です。

#### 例
- OpenAI API 利用額が急増
- Firestore read/write が急増
- Storage 使用量が急増
- Cloud Functions 実行回数が急増
- Map API の請求が急増

#### 影響
- 収益性悪化
- サービス継続性低下
- 意図しない負荷の可能性

---

### 2.8 Security rules misconfiguration
Security Rules の誤設定です。

#### 例
- Viewer が編集できてしまう
- 他人のノートが読める
- 削除権限が広がる
- Storage が不正アクセス可能になる

#### 影響
- 重大な情報漏えい
- 不正操作
- 法務・信用リスク

---

### 2.9 Data deletion issue
データ削除の不具合です。

#### 例
- ノート削除が反映されない
- 写真だけ残る
- 共有カードが残る
- 誤って広範囲削除される

#### 影響
- データ残留または消失
- 重大な顧客対応が必要

---

### 2.10 Subscription entitlement issue
サブスク権限の不整合です。

#### 例
- 購入済みなのに利用できない
- 未購入なのに有料機能が使える
- Apple / Google の購入状態と app_config が不一致
- 価格変更後に表示がずれる

#### 影響
- 売上損失
- 返金・問い合わせ増加
- ストア整合性の問題

---

### 2.11 App crash spike
クラッシュ急増です。

#### 例
- 特定画面でクラッシュ
- 特定 OS で起動不能
- 写真選択で落ちる
- 共有カード生成画面で落ちる

#### 影響
- 利用継続不能
- レビュー悪化
- 離脱増加

---

### 2.12 Store review issue
ストア審査・レビュー問題です。

#### 例
- 審査リジェクト
- プライバシー説明不足
- 権限説明不足
- 課金説明不整合
- 評価低下の継続

#### 影響
- リリース遅延
- 公開停止
- 信頼低下

---

## 3. Severity定義

### 3.1 Severityレベル

| Severity | 定義 | 例 |
|---|---|---|
| S1 Info | 影響小、監視継続で十分 | 軽微な失敗率上昇 |
| S2 Warning | 放置で悪化の可能性あり | 一部機能失敗率上昇 |
| S3 Critical | ユーザー影響またはコスト影響が明確 | ノート保存失敗増加、AI失敗連発 |
| S4 Emergency | セキュリティ事故、データ損失、広範囲停止 | 権限漏えい、誤削除、課金暴走 |

### 3.2 判定基準
以下の観点で Severity を決めます。

- 影響ユーザー数
- 影響機能の重要度
- 継続時間
- 復旧可能性
- データ損失有無
- セキュリティ影響有無
- コスト影響有無

### 3.3 即時 S4 とする条件
次のいずれかで S4 とします。

- 他人のノートや写真が見える
- 削除が誤って広がっている
- 課金が二重計上している疑い
- OpenAI / Storage / Functions コストが異常暴騰している
- Security Rules が破られている
- 未承認の重要設定変更が検知された

---

## 4. 初動対応

### 4.1 基本方針
障害発見時は、まず被害拡大を止めます。  
原因究明より先に、利用継続可否を判断します。

### 4.2 初動手順
1. インシデントを起票する
2. 影響範囲を確認する
3. Severity を仮決定する
4. 直近の変更を確認する
5. ログとメトリクスを確認する
6. 被害拡大を止める措置を検討する
7. 必要なら機能フラグを切る
8. 必要なら緊急停止を有効化する
9. 人間管理者へ通知する
10. ユーザー向け案内の要否を判断する

### 4.3 30分以内に確認するもの
- Crashlytics
- Cloud Functions logs
- Firestore / Storage エラー
- OpenAI 利用状況
- app_config の変更履歴
- Security Rules 変更履歴
- 課金・購入エラー
- サポート問い合わせ急増

### 4.4 初動でやってはいけないこと
- 原因不明のまま本番デプロイする
- AI の提案だけで機能削除する
- 監査なしでデータを消す
- 権限事故の可能性があるのに様子見する
- 課金不整合を放置する

## 5. AI Opsの役割

AI Ops は「分析」「要約」「提案」までを担当します。  
**実行権限は持ちません。**

### 5.1 AI Ops が行ってよいこと
- 異常検知の要約
- 影響範囲の整理
- 原因候補の提示
- 変更候補の提案
- 問い合わせ分類
- 返信ドラフト作成
- 日次・週次レポート作成
- コスト増要因の仮説提示
- 緊急停止候補の提示

### 5.2 AI Ops が行ってはいけないこと
- 本番コード変更
- 本番デプロイ
- Security Rules 変更の確定
- 課金仕様変更の確定
- データ削除
- ユーザー制裁
- 緊急停止解除
- プライバシーポリシー変更

### 5.3 AI Ops の入力制約
- 写真本体は送らない
- 精密GPSは送らない
- 招待トークンは送らない
- 認証情報は送らない
- 個人メモ全文は原則送らない
- 必要最小限の集約データのみ使う

### 5.4 AI Ops の出力形式
- 要点サマリー
- 影響機能
- 可能性の高い原因
- すぐやるべきこと
- 承認が必要な変更
- ユーザー向け案内文案

---

## 6. 人間管理者の役割

### 6.1 役割
人間管理者は最終判断者です。

- 障害の継続可否判断
- 緊急停止判断
- 価格・プラン・制限変更の承認
- データ削除承認
- ユーザー制裁判断
- 本番反映判断
- ストア提出判断

### 6.2 人間だけが実行できる操作
- 本番デプロイ
- 課金仕様変更
- Security Rules 変更
- データ削除
- ユーザー制裁
- プライバシーポリシー変更
- 緊急停止解除
- App Store / Google Play 提出

### 6.3 日常確認項目
- Crashlytics
- Cloud Functions エラー
- OpenAI / Firebase / Storage / Map のコスト
- サブスク整合
- 問い合わせ
- レビュー
- 緊急停止状態
- 監査ログ

## 7. Escalation

### 7.1 エスカレーション基準
以下でエスカレーションします。

- S3 以上
- 15分以上継続する異常
- 影響ユーザー数が増加
- コスト異常が加速
- 権限・削除・課金に関わる疑い
- AI Ops が Emergency 提案を出した

### 7.2 エスカレーション先
1. 運用担当
2. 技術担当
3. 承認者
4. 法務・責任者
5. 必要に応じてサポート担当

### 7.3 緊急時の優先順位
1. セキュリティ
2. データ保全
3. 課金整合
4. 主要機能復旧
5. UX改善

### 7.4 緊急停止の扱い
緊急停止は以下を止めるために使います。

- AI生成
- 共有カード生成
- 招待発行
- 高コスト機能
- 変更系の一部処理

解除は Human Approval 必須です。

---

## 8. Communication

### 8.1 内部連絡
内部連絡では、次を明示します。

- 何が起きたか
- いつから起きたか
- 影響範囲
- 現在の Severity
- 暫定対応
- 必要な承認
- 次回更新時刻

### 8.2 外部連絡
ユーザー案内が必要な場合は、以下を簡潔に伝えます。

- 一部機能に障害があること
- 現在対応中であること
- データ保全の有無
- 復旧見込み
- 回避策があるかどうか

### 8.3 外部連絡で出してはいけない情報
- 内部実装詳細
- 秘密情報
- 認証情報
- 精密位置情報
- 他人の個人情報
- 監査上不要な内部ログ

### 8.4 連絡テンプレの例
- 「現在、写真アップロードに不具合が発生しています。復旧対応中です。」
- 「一部の共有ノートで権限表示に問題が確認されました。安全確保のため関連機能を一時停止しています。」
- 「AI生成機能に遅延が発生しています。ノート作成自体は継続可能です。」

## 9. Rollback

### 9.1 ロールバック対象
ロールバックは以下を対象にします。

- 本番アプリ配信
- Cloud Functions
- Firestore app_config
- Feature Flag
- 共有カードテンプレ
- AIプロンプト
- Map / Geocoding 設定

### 9.2 ロールバック判断
以下でロールバック候補とします。

- リリース直後に crash spike
- 認証失敗増加
- 保存失敗増加
- 権限エラー急増
- 課金不整合
- コスト急増

### 9.3 ロールバック手順
1. 直前変更を特定する
2. 影響範囲を確認する
3. 変更停止を行う
4. 必要なら前バージョンへ戻す
5. 監視を強化する
6. 再発防止策を記録する

### 9.4 ロールバック時の注意
- データ削除の巻き戻しは別問題
- 課金状態の巻き戻しはストア整合を確認
- Security Rules の戻しは慎重に行う
- AI関数の停止だけで済む場合は、全体ロールバックしない

---

## 10. Postmortem

### 10.1 実施条件
S3 以上、またはユーザー影響が大きい障害では postmortem を実施します。

### 10.2 記載内容
- 発生日時
- 検知方法
- 影響範囲
- 根本原因
- 暫定対応
- 恒久対応
- 再発防止策
- 学び
- 未解決事項

### 10.3 重要方針
- 責任追及より再発防止
- 事実と推測を分ける
- AI Ops の分析は参考情報として扱う
- 監査ログと一致する事実を優先する

---

## 11. 再発防止

### 11.1 技術対策
- Security Rules テスト強化
- 主要機能の監視閾値見直し
- 失敗時フォールバック整備
- 再試行制御
- キャッシュ強化
- 削除ジョブの冪等化
- 共有カード生成の検証強化

### 11.2 運用対策
- 承認フロー見直し
- 緊急停止手順の定期訓練
- 問い合わせテンプレ更新
- 週次でのコストレビュー
- 価格・プラン整合確認

### 11.3 AI Ops 対策
- 異常分類プロンプト改善
- レポートテンプレ固定
- 自動実行禁止の再確認
- 危険提案には Human Approval を必須化

---

## 12. Dashboard操作

### 12.1 管理画面
管理画面は Streamlit を使用してよいです。  
ただし、管理画面は **認証必須** であり、危険操作は承認が必要です。

### 12.2 変更可能な項目
コード変更なしで、以下の管理が可能であるべきです。

- サブスク価格の表示値
- プラン範囲
- AI回数制限
- ストレージ上限
- 機能フラグ
- 緊急停止
- 文言の一部
- しきい値の一部

### 12.3 変更してよい範囲
- `app_config` の安全な値
- Feature Flag
- 制限値
- ダッシュボード上の表示設定

### 12.4 変更してはいけない範囲
- 本番デプロイ
- 課金仕様変更の確定
- Security Rules 変更
- データ削除
- ユーザー制裁
- プライバシーポリシー変更

### 12.5 ダッシュボード操作時の注意
- 変更前後の差分を保存する
- 承認者を記録する
- 緊急停止の変更理由を残す
- AI の提案をそのまま反映しない

---

## 13. 禁止事項

以下は禁止です。

- AI が本番コードを勝手に変更する
- AI が本番デプロイする
- AI がユーザーデータを削除する
- AI が課金方針を変更する
- AI が Security Rules を変更する
- 承認なしに緊急停止を解除する
- 監査ログなしで設定変更する
- 個人データを不用意に AI へ送る
- 障害中に原因不明の大規模変更を行う
- 権限事故を軽視する

---

## 14. 未決定事項

以下は運用開始前または追加設計が必要です。

- 緊急停止の解除条件の詳細
- 価格変更の承認フローの厳密な担当者
- Streamlit 管理画面の最終認証方式
- サブスク商品IDと app_config の同期方式
- 監査ログの保存期間
- ユーザー制裁の具体的基準
- 障害時の外部告知テンプレート
- ストアレビュー対応の一次責任者
- 重大障害時の電話連絡体制

必要であれば次に、これを **「実運用向けの SOP 形式」** にして、  
- 障害検知  
- 初動  
- 緊急停止  
- 復旧  
- 事後報告  
までをチェックリスト化した版も作れます。
