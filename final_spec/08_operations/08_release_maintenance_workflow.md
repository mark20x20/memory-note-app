# Release Maintenance Workflow v2

## 1. 目的

Memory Note App / 思い出ノートアプリ のリリース後に、  
**安全に保守し、継続改善し、コストと品質を安定運用するためのワークフロー**を定義する。

このワークフローの前提は以下です。

- AIは**監視・分析・提案・ドラフト作成**まで
- 人間は**承認・実行・最終判断**を行う
- 危険操作は自動化しない
- 本番コード変更、本番デプロイ、課金変更、データ削除、Security Rules変更は必ず承認を通す
- Streamlit 管理画面で済む変更と、コード変更が必要な変更を明確に分ける
- Firebase / OpenAI / Map API / Store課金 / Storage / Cloud Functions の状態を横断監視する

---

## 2. リリースサイクル

### 2.1 基本サイクル
リリース後の運用は、以下の周期で回す。

| 周期 | 主目的 | 主要アウトプット |
|---|---|---|
| 日次 | 異常検知、コスト監視、問い合わせ確認 | 日次運用レポート |
| 週次 | 改善候補の選定、設定見直し | 週次改善案、バックログ更新 |
| 月次 | 収益性、継続率、プラン整合の確認 | 月次ビジネスレビュー |
| リリース前 | 変更内容の検証と承認 | Release Checklist |
| 障害時 | 被害拡大防止と復旧 | Incident Report / Rollback |

### 2.2 リリース単位
- **Release v1.x**
  - 小改善、文言調整、設定変更、制限調整
- **Release v2+**
  - 新機能追加、権限や課金に関わる構造変更
- **Hotfix**
  - 障害修正、緊急停止解除後の安全復旧

### 2.3 運用方針
- 変更は「提案 → 人間レビュー → Staging検証 → Production反映」の順
- 重要変更は **1回で複数をまとめすぎない**
- 価格、権限、削除、規約に触れる変更は分離して扱う
- AI提案は採用候補であって、確定値ではない

## 3. 変更種別

以下の変更種別で運用する。

### 3.1 config change
`app_config`、`remote config` 相当の設定変更。

**例**
- デフォルト文言
- 表示順
- しきい値
- 機能のON/OFF補助値
- AI回数制限の軽微調整
- ストレージ上限の微調整

### 3.2 pricing display change
アプリ内の価格表示やプラン表示の変更。

**例**
- 月額表示の文言変更
- 無料 / 有料プランの案内文変更
- 割引訴求の見せ方変更
- 価格比較表の修正

### 3.3 entitlement change
課金権限やプラン付与内容の変更。

**例**
- どの機能を free / plus / premium に含めるか
- AI回数上限
- Storage 上限
- 共有ノート数の上限
- エクスポート機能の有無

### 3.4 feature flag change
機能の有効/無効切り替え。

**例**
- AI再生成ON/OFF
- 共有カードテンプレ追加
- On This Day ON/OFF
- 高コスト機能の一時停止

### 3.5 prompt version change
AIプロンプトや生成ルールの更新。

**例**
- タイトル生成プロンプト改訂
- 日記生成文体変更
- フォールバック文変更
- ガードレール強化

### 3.6 backend function change
Cloud Functions の修正。

**例**
- AI送信整形
- 削除ジョブ修正
- 招待処理修正
- コスト制御
- ログ改善

### 3.7 mobile app release
React Native + Expo アプリ本体のリリース。

**例**
- UI改善
- バグ修正
- 新画面追加
- 権限制御UI更新

### 3.8 emergency disable
障害やコスト異常時に機能を即時停止する操作。

**例**
- AI生成停止
- 共有カード停止
- 招待停止
- 高コスト機能停止
- 新規作成一時停止

## 4. 変更承認フロー

### 4.1 標準フロー
すべての変更は以下の流れで扱う。

1. **AI Opsが変更案を作成**
2. **人間がレビュー**
3. **必要なら追加調査**
4. **Staging検証**
5. **承認**
6. **Production反映**
7. **監査ログ記録**
8. **反映後監視**

### 4.2 承認区分
| 変更種別 | AI提案 | 人間レビュー | Human Approval | 反映実行 |
|---|---:|---:|---:|---:|
| config change | 可 | 必須 | 条件付き | 管理画面またはFunctions |
| pricing display change | 可 | 必須 | 必須 | 管理画面 |
| entitlement change | 可 | 必須 | 必須 | 管理画面 + ストア整合確認 |
| feature flag change | 可 | 必須 | 条件付き | 管理画面 |
| prompt version change | 可 | 必須 | 必須 | Dashboard/Functions |
| backend function change | 可 | 必須 | 必須 | CI/CD |
| mobile app release | 可 | 必須 | 必須 | CI/CD / Store提出 |
| emergency disable | 可 | 必須 | 必須 | Dashboard 즉時反映 |

### 4.3 承認時に必ず確認すること
- 影響範囲
- ロールバック方法
- ストア整合
- 課金整合
- Security Rules 影響
- データ削除影響
- コスト影響
- ログ監査方法

### 4.4 危険操作の扱い
以下は必ず **Human Approval 必須**。

- 本番デプロイ
- 課金仕様変更
- Security Rules 変更
- データ削除
- ユーザー制裁
- プライバシーポリシー変更
- 緊急停止解除

## 5. AI Opsによる変更提案

### 5.1 AIが担当すること
AI Ops は以下を行う。

- 異常検知の要約
- 変更候補の抽出
- 設定値の見直し提案
- 回数制限の提案
- コスト削減提案
- 失敗率の高い機能の特定
- プロンプト改善案の作成
- サポート返信ドラフトの作成
- リリース前リスク要約
- 週次/月次レポートの作成

### 5.2 AIが出すべき提案形式
各提案は以下の形式にする。

- 変更対象
- 変更理由
- 影響範囲
- 期待効果
- リスク
- ロールバック方法
- 承認要否
- 実行方法
- 参照ログ

### 5.3 AI提案の例
- 「OpenAI 再生成回数が多いため、daily limit を 20→12 に下げる提案」
- 「共有カード生成失敗率上昇のため、Map-first テンプレを一時停止する提案」
- 「Storage 増加が急なため、原本保存関連フラグをOFFにする提案」
- 「タイトル生成の失敗率改善のため、プロンプト v3 に更新する提案」

### 5.4 AI提案の制約
- 本番確定はしない
- データ削除は提案のみ
- ユーザー制裁は提案のみ
- 課金仕様変更は提案のみ
- Security Rules 変更案は提案のみ
- プライバシーポリシー改定案は提案のみ

## 6. 人間レビュー

### 6.1 レビュー責務
人間は、AI提案をそのまま通さず、以下を確認する。

- 本当に必要か
- 影響が大きすぎないか
- ストア整合が崩れないか
- 課金の説明と一致するか
- 権限や公開範囲が変わらないか
- 既存ユーザーを壊さないか
- ロールバック可能か

### 6.2 レビューの観点
| 観点 | 確認内容 |
|---|---|
| 事業 | 売上、継続率、解約率に効くか |
| 技術 | バグや負荷を増やさないか |
| コスト | OpenAI / Storage / Functions が増えすぎないか |
| UX | ユーザー体験が悪化しないか |
| セキュリティ | 権限や公開範囲が危険でないか |
| 法務 | 規約・プライバシーに影響しないか |

### 6.3 レビュー結果
- **Approve**
- **Needs Revision**
- **Reject**
- **Escalate**

## 7. Staging検証

### 7.1 検証目的
本番反映前に、変更が安全に動くか確認する。

### 7.2 検証対象
- config change
- pricing display change
- entitlement change
- feature flag change
- prompt version change
- backend function change
- mobile app release

### 7.3 検証項目
- ログイン
- ノート作成
- 写真アップロード
- AI生成
- 共有カード
- 招待
- 権限表示
- 削除導線
- コスト反映
- ストア価格整合

### 7.4 検証結果の記録
- 変更前後の差分
- テスト結果
- 失敗したケース
- 再現条件
- 修正要否
- 承認可否

### 7.5 Stagingで止めるべきもの
- 権限事故の兆候
- 課金表示の不一致
- AI送信範囲の逸脱
- 共有カードの位置情報漏れ
- Storage削除の不整合

## 8. Production反映

### 8.1 反映の基本
本番反映は、必ず承認後に行う。

### 8.2 反映順
1. 変更内容確定
2. 承認確認
3. 反映前バックアップ
4. Stagingとの差分再確認
5. Production反映
6. 監視強化
7. 反映結果記録

### 8.3 反映手段
- Streamlit Dashboard での `app_config` 更新
- Cloud Functions デプロイ
- React Native + Expo App リリース
- App Store Connect / Play Console 更新

### 8.4 反映後に見るもの
- Crashlytics
- Cloud Functions logs
- Analytics
- コスト
- 問い合わせ
- 課金整合
- 監査ログ

---

## 9. Rollback

### 9.1 ロールバック可能対象
- `app_config`
- feature flags
- prompt versions
- Functions
- mobile app release
- 共有カードテンプレ
- AI文面
- 一部設定値

### 9.2 ロールバック不能または慎重な対象
- 既に削除されたデータ
- 既にストアに反映済みの課金商品
- Security Rules の広範変更
- 既に配布済みの課金プラン説明
- プライバシーポリシー

### 9.3 ロールバック手順
1. 問題検知
2. 影響範囲確認
3. 変更停止
4. 直前の安定版へ戻す
5. 監視強化
6. 再発防止策作成

### 9.4 緊急時の優先
- まず停止
- 次に安全な代替へ戻す
- 最後に根本修正

## 10. Streamlit Dashboardで行う変更

以下は **コード変更なし** で運用可能なもの。

### 10.1 変更可能な領域

- feature flags
- 価格表示文言
- プラン案内文言
- AI回数制限
- Storage上限
- 一部の通知文言
- 緊急停止フラグ

- フォールバック文言

### 10.2 具体例
- 「AI再生成は1日10回まで」
- 「freeプランは共有カード保存のみ可」
- 「premiumプランのStorage上限を10GBへ」

- production では二重確認
- 反映後に即時監視

## 11. コード変更が必要な変更

以下はダッシュボードだけでは完結しない。

### 11.1 例
- 新画面追加
- 新API追加
- 共有カードレイアウトの変更
- AI処理のロジック変更
- Storage処理の方式変更
- Security Rules の変更
- 課金フロー変更
- 権限モデル変更
- 新機能実装
- バグ修正の本体実装

### 11.2 運用方法
1. Issue 化
2. AIが実装案を作成
3. 人間が要件確認
4. 開発ブランチで実装
5. テスト
6. Staging確認
7. 承認
8. 本番反映

### 11.3 注意
- コード変更は設定変更と混ぜない
- 危険な変更は別PRにする
- 本番デプロイは必ず人間承認

## 13. Release Checklist

### 13.1 リリース前チェック
- [ ] 変更内容が承認済み
- [ ] Staging検証済み
- [ ] Crashlytics で新規重大クラッシュなし
- [ ] Functions エラー率が許容内
- [ ] OpenAI 利用量が正常
- [ ] Storage 増加が異常でない
- [ ] 課金商品IDと表示が一致
- [ ] Security Rules 変更なし、または承認済み
- [ ] プライバシーポリシー変更なし
- [ ] 緊急停止フラグの状態確認
- [ ] ロールバック手順が用意済み
- [ ] 監査ログ出力確認
- [ ] サポート案内準備済み

### 13.2 リリース後チェック
- [ ] 起動成功率
- [ ] ログイン成功率
- [ ] ノート作成成功率
- [ ] 写真アップロード成功率
- [ ] AI生成成功率

- [ ] 課金導線異常なし
- [ ] 権限エラー異常なし
- [ ] コスト異常なし
- [ ] 問い合わせ急増なし

## 14. Maintenance Backlog

### 14.1 バックログ分類
| 種別 | 例 |
|---|---|
| Bug | 保存失敗、表示崩れ、権限エラー |
| Cost | OpenAIコスト増、Storage増 |
| UX | 文言改善、導線改善 |
| Reliability | 再試行、フォールバック強化 |
| Security | 権限見直し、削除導線改善 |
| Growth | 共有導線、SNSカード改善 |
| Ops | 監視改善、ダッシュボード改善 |

### 14.2 優先度
- P0: 障害・課金・権限・削除・漏えい
- P1: 重要UX、コスト改善、再発防止
- P2: 改善、拡張、実験

### 14.3 バックログ更新元
- AI Ops レポート
- 問い合わせ
- ストアレビュー
- 障害後振り返り
- コストレポート
- 週次レビュー

## 15. Weekly Improvement Loop

### 15.1 週次改善ループ
毎週以下を回す。

1. AI Ops 週次レポート確認
2. 問い合わせ分類確認
3. コスト傾向確認
4. 失敗率上位を確認
5. 改善候補を選定
6. 影響度を見積もる
7. 変更種別を分ける
8. Human Approval が必要か判定
9. Stagingに送る
10. Production反映
11. 反映後の効果測定

### 15.2 週次で見る指標
- ノート作成成功率
- AI生成成功率
- 共有カード生成率
- 写真アップロード失敗率
- 権限エラー率
- 退会率
- 継続率
- 問い合わせ件数
- コスト推移

### 15.3 週次の改善テーマ例
- AIプロンプトの微修正
- 失敗時文言の改善
- Storage圧縮率の調整
- 検索UX改善
- 共有カードテンプレ調整
- FAQ更新
- 回数制限調整

## 16. Monthly Business Review

### 16.1 月次レビューの目的
ビジネスとして継続可能かを判断する。

### 16.2 見る指標
- 売上
- 解約率
- プラン別ARPU
- AIコスト
- Storageコスト
- Functionsコスト
- Map APIコスト
- 課金失敗率
- ストアレビュー
- 継続率
- 主要機能利用率

### 16.3 月次で決めること
- 価格の妥当性
- プラン範囲の見直し
- 無料枠の妥当性
- 高コスト機能の継続可否
- 有料機能の訴求修正
- 次月の重点改善テーマ

### 16.4 注意
- 価格変更は即決しない
- ストア整合を先に確認
- ユーザー説明文も同時に見直す
- 必ず変更履歴を残す
